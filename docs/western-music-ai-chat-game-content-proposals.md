# 洋楽AIチャット向け「ゲーム感覚で詳しくなる」コンテンツ案

チャット上の議論・既存の設計メモ（`docs/room-social-music-games-plan.md`）を踏まえ、**実装の軽さと体験のバランス**ごとに整理した案の置き場です。実装タスクではなく、企画・優先度検討用です。

---

## 1. プロダクトの強み（企画の前提）

- **同期視聴**：曲が流れていることがコンテンツの中心になる。
- **同一文脈にある機能**：曲解説、豆知識、（方針次第で）曲解説後クイズ、お題プレイリスト／ミッションなど。
- ゲーム感は **正解主義より「短時間・全員参加・外しても笑える」** に寄せると、初対面ルームでも回りやすい（`room-social-music-games-plan.md` の設計原則と整合）。

---

## 2. 企画の軸（三本柱）

| 柱 | 内容 | ユーザーが「詳しくなる」方向 |
|----|------|------------------------------|
| **耳ゲー** | 今流れている音から分かることに焦点 | 年代・質感・楽器・テンポ感などの**聴き方の語彙**が増える |
| **偏愛ゲー** | 正解より「なぜこの曲か」がコンテンツ | **接続の理由・文脈**で洋楽との距離が縮む |
| **協力ゲー** | お題に沿って集合知でリストが育つ | **テーマと曲の関係**の引き出しが鍛えられる（既存お題ミッションと親和） |

---

## 3. 案一覧（実装・運用の軽さ目安）

### 3.1 ルールだけで試せる（実装薄め）

| 案 | 進行のイメージ | メリット | 注意 |
|----|----------------|----------|------|
| **30秒スタイル投票** | 粗い軸（例：シンセPop寄り／ロック寄り／その他）で投票 → 多数派や拍手で締め | 全員参加・短時間・誤情報リスクが低い | AIは「言い換え・まとめ」に寄せ、断定しすぎない |
| **次の一曲当て（正解なし）** | ヒントだけ出して各自がURL投げ → 拍手採用など | 選曲UIと直結、正解不要で盛り上がりやすい | 司会のテンポだけ決める |
| **偏愛1曲＋質問1つ** | 各自1曲＋一言、他者は質問1つまで | 対人距離が縮みやすい | 時間上限を決めるとよい |
| **曲解説から1問だけ（司会クイズ）** | ホストが曲解説テキストから**自分で1問**作る、AIは補助 | 幻覚・誤出題をコントロールしやすい | ホスト負荷あり |

### 3.2 プロダクトに載せると強い（中〜重）

| 案 | 内容 | 既存・近傍との相性 |
|----|------|-------------------|
| **曲解説後三択（低頻度）** | 曲解説のあと、同じ曲の文脈で**たまに**3択1問 | `room-social-music-games-plan.md` §8 の具体案・出題ゲート・設問方針と整合 |
| **部屋デイリーチャレンジ** | その日のテーマを部屋で共有し、**数曲だけ**協力で埋める | お題・ミッション思想の拡張として説明しやすい |
| **ミニ称号（非ランキング）** | 「変な接続で次曲を繋いだ」など**笑い系の表彰** | マウント文化を避けつつ参加動機を足す |

### 3.3 最初の1本としておすすめのパッケージ

**「耳投票 → 曲解説 →（たまに）三択」**の順。

1. **耳投票**：全員参加でテンポが上がり、語彙が揃いやすい。  
2. **曲解説**：同じ曲の文脈で情報が整理される。  
3. **三択**：**毎回ではなく**出す前提で、コスト・リピート感・誤出題リスクを抑える（§8.1b の「未出題より誤出題」方針と整合）。

---

## 4. 仕様化・優先度を決めるときのチェックリスト

- [ ] 1ラウンド **30〜90秒**程度で切れるか。  
- [ ] **ROM勢・途中入室**が居心地悪くないか（任意参加・スキップ）。  
- [ ] **誤情報**が刺さらない設問か（断定・歌詞引用・出典曖昧な領域）。  
- [ ] **AIの役割**は「素材生成」「司会補助」「オフ」のどれか（判定で揉めないか）。  
- [ ] **ホスト権限**（スキップ、難易度、オンオフ）との関係。

---

## 5. 主軸ラベル（実装方針の分岐）

| ラベル | 説明 |
|--------|------|
| **A. 軽いミニゲーム** | 既存ルームに「今だけオン」で乗る短い遊び（ルール＋チャット運用中心でも可）。 |
| **B. 本格クイズモード** | 出題・採点・ラウンド管理などUI／状態が厚め。 |
| **C. AI主導の自動出題** | 生成中心。品質・コスト・誤答対策の設計が重い。 |

**推奨の進め方**：まず **A** で反応を見て、伸びた領域だけ **B** や **C** に落とす（`room-social-music-games-plan.md` §6 と同趣旨）。

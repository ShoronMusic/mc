# 運用と料金の注意事項（洋楽AIチャット）

開発〜一般開放を見据えた、環境変数・管理画面・料金・広告まわりの整理です。

---

## 1. 秘密情報（`.env.local`）

| 項目 | 注意 |
|------|------|
| **Git** | `.env*.local` は `.gitignore` 済み。**まだコミットしていないなら今後も含めない。** 一度でもコミットしたら履歴に残るため、キーはローテーション推奨。 |
| **共有** | チャット・スクリーンショットに **API キーや Service Role を貼らない。** |
| **Vercel** | 本番は **ダッシュボードの Environment Variables** に同じ名前で登録。リポジトリに `.env.local` は上げない。 |

## 2. Supabase と Gemini 利用ログ（自前ダッシュボード）

| 項目 | 内容 |
|------|------|
| **テーブル** | `gemini_usage_logs` を SQL Editor で作成（手順は `docs/supabase-gemini-usage-logs-table.md`）。 |
| **保存条件** | **`SUPABASE_SERVICE_ROLE_KEY`** があるときのみ DB に INSERT。`GEMINI_USAGE_PERSIST=0` で保存オフ。 |
| **コンソールだけ** | `GEMINI_LOG_USAGE=1` でサーバーログに JSON 1 行（DB 不要）。 |

### 管理画面 `/admin/gemini-usage`

| 項目 | 内容 |
|------|------|
| **誰が見られるか** | **`STYLE_ADMIN_USER_IDS`** に、Supabase **Authentication → Users の User UID** をカンマ区切りで列挙。表示名（小龍さんなど）と **ログインアカウントの UUID は別**。 |
| **403 のとき** | ① **Google でログインしているか**（ゲスト表示名だけでは未ログインのことがある）② **UID がリストに入っているか** ③ **サーバー再起動**後か。 |
| **API の仕様** | Route Handler では `getSession()` が空になりがちなため、**`getUser()` で認証**する実装にしている。 |

### ログの見方（自前）

- **入力トークン** … プロンプト・履歴の量（料金の入力単価に比例）
- **出力トークン** … AI 返答の量（出力単価に比例）
- **context** … 機能別（`chat_reply` / `tidbit` / `comment_pack_base` など）
- **Google の請求額** … **必ず [Gemini API の利用額](https://aistudio.google.com/usage) で確認**。自前ログは「内訳把握・試算」用で、請求の正は Google 側。

## 3. 料金の目安

- **自前の「入力トークン合計」だけでは請求額は確定しない**（無料枠・キャッシュ単価・モデル差など）。
- **ざっくり試算**（例: Gemini 2.5 Flash 有料ティアの目安単価）は  
  [公式料金ページ](https://ai.google.dev/gemini-api/docs/pricing) の **1M トークンあたり単価 × 自前ログの合計** で概算可能。
- Google ダッシュボードの **「1か月の費用の上限」進捗**と、**期間フィルタした合計**は **別の数字**になり得る（更新遅延最大 24 時間などもある）。

---

## 4. 一般開放を見据えたコストの考え方

- 料金は **ルーム数より「API 呼び出し回数 × トークン」** で決まる。
- **重い要因の例**
  - **1 曲あたり comment-pack** … 基本 1 回 + 自由 3 回 = **Gemini 約 4 回/曲**
  - **tidbit（無発言時）** … ルームが静かなほど **回数が増えやすい**
  - **チャット** … 履歴が長いほど入力トークン増
- **10 ルームでも利用パターンで 1 日の料金は桁が変わる**ため、一般公開前に **tidbit 間隔・1 日あたり AI 上限・再利用（DB）** などの設計が有効。

## 5. マネタイズ・広告（準備のメモ）

| 項目 | メモ |
|------|------|
| **Google AdSense** | **ログイン必須のチャットだけ**だと審査・単価が厳しめになりがち。**ランディング・記事・曲紹介**などコンテンツページとセットで検討しやすい。 |
| **API 費用とのバランス** | 広告だけで API 代を全額カバーできるかは **PV・単価次第**。先に **月次/日次の API 予算上限**（Google 側の上限 + アプリ側制限）を用意すると安全。 |
| **ポリシー** | 利用規約・**プライバシーポリシー**、必要に応じて **Cookie 同意**（EU 等）を広告導入前に整理。 |

## 6. コメントの DB 蓄積と再利用（comment-pack）

| 項目 | 内容 |
|------|------|
| **初回** | その `video_id` で comment-pack を実行すると AI が 4 本生成し、`song_tidbits` に **ai_commentary / ai_chat_1〜3** として保存される。 |
| **2回目以降** | 同じ `video_id` で **4 本そろっていれば AI を呼ばず** DB から返す（**Gemini 0 回**）。画面上は **[DB]** プレフィックス。 |
| **条件** | キャッシュ読み取りに **`SUPABASE_SERVICE_ROLE_KEY` 推奨**（無いと RLS で他ユーザー再生時にヒットしない場合あり）。 |
| **強制新規** | `COMMENT_PACK_SKIP_CACHE=1` で常に AI 生成。 |

豆知識 API（`/api/ai/tidbit`）はもともと **`tidbit_library` を先に検索**してから AI、という流れで、曲単位の蓄積が進むほど **DB から出る割合**が増えていきます。

## 7. UX まわり（実装済みメモ）

- **comment-pack 後に同じ曲の豆知識が続くと内容が被る**ため、comment-pack を出した動画では **次の曲まで「その曲向け tidbit」を抑え、一般豆知識に寄せる**処理を入れている（同期ルーム側）。

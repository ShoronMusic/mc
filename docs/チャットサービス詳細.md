# チャットサービス 詳細仕様

Music8 Lounge のチャット機能について、役割・技術・プロトコル・DB連携・運用をまとめる。

---

## 1. 概要と目的

| 項目 | 内容 |
|------|------|
| **役割** | 同一ルーム内の参加者どうしの会話、および AI の相槌・解説・進行のやり取りを行う。 |
| **前提** | ルーム（またはセッション）ごとにチャット・再生が分離する。MVP ではルーム数を1にすることも可。**一人でAIと楽しむ**・**同時に複数利用**を見据え、設計上は複数ルーム／複数セッションの並列を許容する（`docs/同時利用と一人でAIと楽しむ設計.md` 参照）。 |
| **保存** | 会話ログは DB（Supabase）に保存し、振り返り・ユーザー館・改善に利用する。 |

---

## 2. 機能一覧

| 機能 | 説明 |
|------|------|
| **テキスト送受信** | 参加者が短文を送信し、同じルームの全員に即時表示する。 |
| **表示名の表示** | 発言ごとに「誰が」を表示。正規ユーザーは display_name、ゲストはその回の表示名（例: Guest-1234）。 |
| **AI の発言** | 曲解説・相槌・盛り上げを AI が投稿。message_type='ai' で他発言と区別。 |
| **システムメッセージ** | 入室・退室の通知（例: 「〇〇さんが入室しました」）。message_type='system'。 |
| **YouTube URL の扱い** | メッセージ本文に YouTube URL が含まれた場合、曲として解釈し再生を切り替え＋AI 解説をトリガーする。 |
| **タイピング表示（任意）** | 「入力中」を他参加者に表示し、会話の臨場感を出す。 |
| **履歴表示** | 入室時に直近 N 件のメッセージを DB から取得して表示。その後はリアルタイムで追加。 |

---

## 3. 技術構成

### 3.1 リアルタイム基盤の選択

| 方式 | 概要 | 向いている場合 |
|------|------|----------------|
| **Ably** | マネージドの Realtime サービス。チャネルに publish すると購読者に配信。 | Vercel のみで完結させたい。Socket サーバーを運用したくない。 |
| **Socket.io** | 自前の Node サーバーで WebSocket を提供。 | 既存の Node 経験がある。Railway / Render 等でサーバーを1つ立てられる。 |

- **チャットと同期再生**は、同じ基盤で「チャネル（ルーム）ごとにイベント種別で分ける」形が扱いやすい。
  - 例: ルーム `room:main` に対して `chat:message` / `playback:play` / `playback:seek` などを送受信。

### 3.2 チャネル設計（例: Ably）

| チャネル名 | 用途 |
|------------|------|
| `room:main` または `lounge:main` | MVP の1ルーム。チャット・再生イベントをすべてこのチャネルでやり取り。 |

- 将来ルームが増えた場合は `room:{roomId}` で分ける。

### 3.3 メッセージの流れ（概要）

```
[クライアントA] テキスト入力 → 送信
  → アプリで「URL かどうか」判定
  → 通常テキスト: リアルタイムチャネルに publish + API で DB に保存
  → YouTube URL: 曲切り替え処理 + AI 解説トリガー + 上記と同様に publish + DB 保存

[リアルタイムチャネル] を全クライアントが subscribe
  → 受信したメッセージを画面に追加表示

[AI] 曲解説 or 相槌を生成
  → サーバー or バックエンド API がリアルタイムチャネルに publish + DB に保存
  → 全クライアントに表示
```

- DB は「正」とする。リアルタイムは「即時配信」用。障害時は DB から履歴を再取得して復元できるようにする。

## 4. イベント・プロトコル

### 4.1 クライアント → サーバー（または Ably 経由）

| イベント名 | ペイロード例 | 説明 |
|------------|--------------|------|
| `chat:send` | `{ sessionId, userId or guestId, body, displayName }` | テキスト発言を送信。サーバーで検証・DB 保存・配信。 |
| `chat:typing` | `{ sessionId, userId or guestId, displayName }` | 入力中であることを通知（任意）。 |

- **URL 検出**: `body` に YouTube URL が含まれる場合は、送信前にまたはサーバー側で検出し、曲切り替え＋AI 解説のトリガーとする。

### 4.2 サーバー（または Ably）→ クライアント

| イベント名 | ペイロード例 | 説明 |
|------------|--------------|------|
| `chat:message` | `{ id, sessionId, messageType: 'user'|'ai'|'system', userId?, guestId?, displayName?, body, createdAt }` | 1件のメッセージを配信。 |
| `chat:history` | `messages[]` | 入室時などに、直近のメッセージ一覧を返す（API または初回 push）。 |
| `chat:typing` | `{ displayName, userId? }` | 誰かが入力中であることを配信（任意）。 |

- `id`, `createdAt` は DB 保存後にサーバーが付与して配信すると、クライアントと DB の整合が取りやすい。

### 4.3 メッセージ種別（message_type）

| 種別 | 説明 | 表示例 |
|------|------|--------|
| `user` | 参加者（人間）の発言。 | 「〇〇: このサビ最高」 |
| `ai` | AI の曲解説・相槌・盛り上げ。 | 「AI: この曲、〇〇ですよね…」 |
| `system` | 入退室などシステム通知。 | 「〇〇さんが入室しました」 |

## 5. DB 連携

### 5.1 保存するテーブル

- **chat_messages**（`docs/DB設計_会話ログ保存版.md` 参照）
  - `session_id`, `message_type`, `user_id` or `guest_id`, `body`, `created_at`

### 5.2 保存のタイミング

| 発信元 | 流れ |
|--------|------|
| ユーザー発言 | 1) API で受信 → 2) バリデーション（文字数・NGワード等）→ 3) DB に insert → 4) リアルタイムで `chat:message` を配信（DB の id, created_at を含める）。 |
| AI 発言 | 1) 曲解説 or 相槌を生成 → 2) DB に insert → 3) リアルタイムで `chat:message` を配信。 |
| システム | 1) 入退室を検知 → 2) DB に insert（任意。ログに残す場合）→ 3) リアルタイムで配信。 |

- **履歴取得**: 入室時に `session_id` で `chat_messages` を `created_at` 昇順で取得（例: 直近 100 件）。その後はリアルタイムで追加表示。

### 5.3 セッション（会）との対応

- 1回の「会」= 1つの `sessions` レコード。
- その会のチャットはすべて `session_id` で紐づく。参加者一覧は `session_participants`、曲は `session_songs` で管理。

---

## 6. 参加者と表示名

| 参加種別 | 識別子 | 表示名の扱い |
|----------|--------|--------------|
| 正規ユーザー | `user_id`（Supabase Auth 等） | `users.display_name` を表示。 |
| ゲスト | `guest_id` または一時ID | 入室時に入力した名前、または「Guest-xxxx」。 |

- ゲストは同一ブラウザ内では同じ `guest_id`（またはクライアント生成ID）を使い、その会では一貫した表示名で表示する。

---

## 7. 制限・モデレーション

| 項目 | 方針例 |
|------|--------|
| **文字数** | 1発言あたり上限（例: 500 文字）。超えたら送信拒否し、クライアントでエラー表示。 |
| **レート制限** | 同一ユーザー/ゲストで N 件/分 まで（例: 10 件/分）。超えたら 429 やメッセージで注意。 |
| **ゲストの制限** | ゲストは「テキスト送信のみ」「1日〇件まで」など、必要に応じて制限。 |
| **NGワード** | 簡易フィルタ（禁止語リスト）でマスクまたは拒否。管理者が後から確認できるようにログは残す。 |

---

## 8. UI との対応

- **レイアウト**: 上段＝参加者一覧、中段左＝チャット表示、中段右＝YouTube＋Now Playing、下段＝入力欄（プラン通り）。
- **入力欄**: テキスト＋送信ボタン。YouTube URL を貼り付けた場合はそのまま送信してよい（サーバー側で URL 検出し、曲切り替え＋AI 解説を発火）。
- **表示**: メッセージは時系列で下に追加。`message_type` に応じて「ユーザー」「AI」「システム」でスタイルを変える（色・アイコン等）。
- **タイピング表示**: 任意。`chat:typing` を受信したら参加者バーに「〇〇が入力中」を表示し、数秒で消す。

---

## 9. 同期再生との関係

- **同じリアルタイム基盤**（Ably の同じチャネル、または Socket.io の同じ namespace/room）で、
  - チャット: `chat:message`, `chat:typing`
  - 再生: `playback:play`, `playback:pause`, `playback:seek`, `playback:changeVideo`
  のようにイベント名で分けて送受信する。
- クライアントは「チャット用」と「再生用」の両方のイベントを subscribe し、それぞれ UI とプレイヤーに反映する。

---

## 10. まとめ

| 項目 | 内容 |
|------|------|
| **役割** | 同じ会の参加者と AI が、テキストで会話・相槌・解説をやり取りする。 |
| **技術** | Ably または Socket.io でリアルタイム配信。Supabase に会話ログを保存。 |
| **メッセージ** | user / ai / system。YouTube URL は曲切り替え＋AI 解説のトリガー。 |
| **制限** | 文字数・レート制限・ゲスト制限・簡易NGワードで品質を保つ。 |

上記を満たす実装にすれば、チャットサービスとして必要な機能と品質が揃う。

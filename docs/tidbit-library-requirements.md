# 豆知識ライブラリ要件・設計

## 1. 目的

- **ライブラリ化**: チャットで披露した AI の豆知識を保存し、**キーワード（アーティスト・曲名・その他）をキーに検索して再利用**する。
- **混在運用**: 豆知識を披露するタイミングで、
  - **ライブラリにヒット**すれば → ライブラリから選んで披露（**生成 API を叩かない**）
  - **ヒットしなければ** → 従来どおり AI でその場生成し、披露すると同時に**新規登録**してライブラリを充実させる。
- **効果**:
  - チャットの運営が豊かになる（蓄積された豆知識＋その場の一言の両方）。
  - **生成 API の呼び出し量を抑制**し、費用を抑える。
  - ライブラリが洋楽豆知識の「財産」として育っていく。

---

## 2. 用語

| 用語 | 意味 |
|------|------|
| **豆知識** | 30秒無発言時に AI が披露する 1〜2 文の一文。 |
| **ライブラリ** | 過去に披露した（または登録した）豆知識を、キーワードと紐づけて保存した DB 上の一覧。 |
| **その場生成** | ライブラリにヒットしなかったときに、Gemini 等で新規に生成する豆知識。 |

---

## 3. 機能要件

### 3.1 ライブラリへの保存（登録）

- **保存するタイミング**: **その場生成で新しい豆知識を披露した直後**に、その文をライブラリに 1 件登録する。
- **保存する内容**（案）:
  - `body`: 豆知識の本文（テキスト）。
  - `video_id`: 披露時点で流れていた曲の YouTube video_id（任意）。
  - `artist_name`, `song_title`: 披露時点の曲情報（任意。検索キーとして利用）。
  - `keywords`: 検索用キーワード（配列またはカンマ区切り）。例: アーティスト名、曲名、ジャンル、年代。その場のチャットから抽出した語があれば追加。
  - `created_at`: 登録日時。
- **重複**: 本文が**完全一致または極端に類似**する既存レコードがあれば、登録しない（重複抑制）。判定は簡易でよい（本文の正規化＋一致 or 類似度しきい値）。

### 3.2 ライブラリからの選出（検索）

- **検索のタイミング**: 豆知識を披露するタイミング（30秒無発言が満たされたとき）の**最初**に、ライブラリを検索する。
- **検索キー**（優先順の例）:
  1. **現在流れている曲**: `video_id` または `artist_name` / `song_title` が一致するもの。
  2. **直近チャットから抽出したキーワード**: アーティスト名・曲名・ジャンル・年代など。キーワードに一致する `keywords` や `body` を持つレコード。
  3. **汎用**: 曲が流れていない、またはヒットしない場合は「洋楽・音楽一般」用のキーワード（例: 空 or "洋楽"）で検索。
- **選び方**: ヒットした候補のうち、**直近で使っていない**ものを 1 件選ぶ（下記「再利用制御」参照）。複数あればランダムまたは古い登録から選ぶなど、方針を 1 つ決める。

### 3.3 再利用制御（同じ豆知識の連続・多用を防ぐ）

- **同じルーム内**: 直近 N 件の「披露した豆知識の id」をセッションまたは DB で保持し、その id は選ばない。N は 10〜20 程度を想定。
- **同一レコードの再披露**: 少なくとも「直近 1 時間以内に同じ id をこのルームで使った」場合は選ばない、などのルールを設けてもよい。
- これにより、**ライブラリからの発言と、その場生成の発言が自然に混在**し、同じ話が何度も続かないようにする。

### 3.4 披露フロー（ライブラリとその場生成の混在）

1. **トリガー**: 30秒無発言が満たされた（既存ロジック）。
2. **ライブラリ検索**: 現在の曲＋（あれば）直近チャットから抽出したキーワードでライブラリを検索。
3. **分岐**:
   - **ヒットし、かつ再利用制御をパスする 1 件がある**  
     → その豆知識をチャットに投稿。**生成 API は呼ばない。**
   - **ヒットしない、または再利用制御で使える候補が残っていない**  
     → 従来どおり **AI でその場生成** → 披露 → **その文をライブラリに新規登録**（重複チェック付き）。

### 3.5 キーワード抽出（検索・登録の両方で使う）

- **曲情報**: 現在流れている曲がある場合、`artist_name`・`song_title`（または video_id から取得したメタデータ）をそのままキーワードとして利用。
- **チャットからの抽出**: 直近メッセージ本文から、アーティスト名・曲名・ジャンル・年代らしき語を抽出する。  
  - 実装方針の例: 簡易ルール（固有名詞っぽい語を拾う）／Gemini に「キーワードだけ抽出」で短いプロンプトを送る／など。コストと実装コストのバランスで決定。
- 登録時は「その場の曲＋抽出キーワード」を `keywords` や `artist_name` / `song_title` に格納する。

## 4. 非機能要件

| 項目 | 要件 |
|------|------|
| **コスト** | ライブラリヒット率が上がるほど生成 API 呼び出しが減り、月次コストが下がる。ヒット率の目標は運用で見ながら調整（例: 3割→5割→…）。 |
| **レイテンシ** | ライブラリ検索は DB アクセスのみとし、豆知識披露の体感遅延を増やさない。 |
| **品質** | ライブラリの豆知識は「過去に一度披露した」ものなので、そのまま再利用してよいレベルとする。不適切な 1 件が混ざった場合は、運用で非公開フラグや削除で対応する余地を残す。 |

## 5. データ構造（案）

### 5.1 テーブル: `tidbit_library`

| カラム | 型 | 説明 |
|--------|-----|------|
| id | uuid | 主キー |
| body | text | 豆知識の本文。NOT NULL。 |
| video_id | text | 披露時点の曲の YouTube video_id（任意）。 |
| artist_name | text | 披露時点のアーティスト名（任意）。 |
| song_title | text | 披露時点の曲名・タイトル（任意）。 |
| keywords | text[] または text | 検索用キーワード。配列またはカンマ区切り。 |
| created_at | timestamptz | 登録日時。 |
| room_id | text | 登録時にどのルームで披露したか（任意。分析用）。 |

- 検索用インデックス: `video_id`, `artist_name`, `keywords`（GIN または LIKE 用）、`created_at`。

### 5.2 再利用制御用

- **案 A**: ルームごとに「直近で披露した tidbit_library.id のリスト」を **サーバーのメモリ or Redis** に保持。最大 20 件など。
- **案 B**: `tidbit_usage` のような別テーブルで「(tidbit_id, room_id, used_at)」を記録し、「直近 1 時間以内に同じ tidbit_id がこの room_id で使われている」なら選ばない。
- 初期実装は **案 A**（メモリに直近 N 件の id を保持）がシンプル。複数インスタンスで共有する必要が出たら案 B や Redis を検討。

## 6. API・処理の変更点

| 箇所 | 変更内容 |
|------|----------|
| **tidbit 披露のトリガー**（ルーム） | 変更なし。30秒無発言で 1 回、tidbit 用 API を呼ぶ。 |
| **POST /api/ai/tidbit** | ① リクエストで `videoId`・（任意）`recentMessages` を受け取る。② **まずライブラリ検索**（video_id / キーワード、再利用制御を考慮）。③ ヒットすればその `body` を返し、**生成は呼ばない**。④ ヒットしなければ **generateTidbit を呼んで生成** → 返すとともに **ライブラリに 1 件 insert**（重複チェック付き）。 |
| **generateTidbit**（Gemini） | 従来どおり。ライブラリにヒットしなかったときだけ呼ぶ。引数に `recentMessages` を追加するかは「チャットの流れを見ながら」要件に合わせる。 |
| **新規** | ライブラリ検索用の関数・API（内部）。ライブラリ登録用の関数（内部）。 |

---

## 7. 受け入れ条件

- [ ] 豆知識披露時に、**ライブラリに該当するレコードがあれば、その本文が返り、生成 API が呼ばれない**こと。
- [ ] ライブラリにヒットしなかった場合のみ **AI で生成**され、**その本文がライブラリに 1 件登録**されること。
- [ ] 同じルームで短い間に**同じ豆知識が何度も連続して披露されない**こと（再利用制御が効いていること）。
- [ ] 登録済み本文と**ほぼ同一**の新規生成結果は、ライブラリに重複登録されないこと（重複抑制）。

## 8. 今後の拡張（スコープ外）

- 管理画面でライブラリの一覧・検索・非公開化・削除。
- キーワード抽出を Gemini の短いプロンプトで行い、精度を上げる。
- ライブラリの「お気に入り」やユーザー投稿の豆知識の申請・採用。

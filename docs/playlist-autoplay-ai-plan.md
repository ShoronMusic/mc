# YouTube プレイリスト連続再生 + 曲ごと AI 解説（設計メモ）

## 目的

- ユーザーが YouTube のプレイリスト URL を貼るだけで、先頭から順に再生する。
- 各曲ごとに AI 解説を自動投稿する。
- ルーム同期再生（複数参加者）を崩さない。

## 想定ユースケース

- 例: `https://www.youtube.com/playlist?list=...` を投稿
- ルーム内でプレイリストを「再生キュー化」
- 現在曲終了時に次曲へ自動遷移
- 曲開始タイミングで AI 解説を投稿

## 実装方針（高レベル）

1. プレイリスト URL の検証と ID 抽出
2. YouTube Data API で動画 ID 一覧を取得
3. ルーム単位の再生キューを作成
4. 再生完了イベントで次曲へ進行
5. 各曲で AI 解説生成（キャッシュ優先）
6. 末尾到達で終了（または loop 指定時に先頭へ）

## 最小仕様（MVP）

- 1 回の投入で最大 20 曲まで（上限超過は切り捨て）
- 対象は公開プレイリストのみ
- AI 解説は 1 曲につき 1 本（短文）
- 解説失敗時は再生を止めずスキップ
- 手動スキップ / 停止 / キュー破棄を提供

## 先に決めるべきプロダクトルール

- プレイリスト投入権限（全員 / 管理者 / ルーム主のみ）
- 手動選曲が来た時の優先度（キュー一時停止 or 継続）
- 曲間待機秒数（0 / 2 / 5 秒など）
- 1 ルーム同時キュー数（推奨 1）
- 解説表示位置（チャット欄 / 専用欄）

## 懸念事項

### 1) API クォータと取得失敗

- YouTube Data API の日次クォータ枯渇で展開不可になる可能性。
- 非公開/削除動画混在時の欠番処理が必要。
- 対策:
  - 取得失敗時の明確なユーザー通知
  - 上限曲数の導入
  - 再試行回数の制限

### 2) AI コスト増

- 曲数に比例して Gemini 呼び出しが増加する。
- 対策:
  - 既存 comment-pack/解説キャッシュを優先利用
  - 同一動画の再生成を抑制
  - ルームごと/日次の生成上限

### 3) UX（長尺プレイリスト）

- 100 曲超で進行管理が重く、途中参加者が状況を把握しづらい。
- 対策:
  - キュー表示（現在位置 / 残り曲数）
  - 先頭 N 曲のみ取り込み
  - 途中から再開機能（任意）

### 4) 同期再生の競合

- 複数ユーザー操作で「次曲自動遷移」と「手動変更」が衝突する。
- 対策:
  - 進行制御者を 1 人に限定、または投票制
  - 操作優先ルールを明文化（手動優先など）

### 5) 品質と安全性

- メタ情報不足の動画で誤解説が出る可能性。
- 対策:
  - 既存の AI 出力ポリシーを適用
  - 根拠が弱い場合は短い汎用解説へフォールバック
  - エラーパターンをログ化して改善サイクルへ

### 6) 荒らし・運用負荷

- 大量 URL 投稿や不適切プレイリスト投入の懸念。
- 対策:
  - 投入頻度制限
  - 1 投稿あたり最大曲数制限
  - 管理者による強制停止・ブラックリスト

## 非機能要件の観点

- 可観測性:
  - `playlist_started`, `track_advanced`, `ai_comment_posted`, `playlist_stopped` のイベントログを残す
- 障害耐性:
  - AI/YouTube API 失敗時に再生進行を継続できること
- パフォーマンス:
  - キュー展開はバックグラウンド化し、UI ブロックを避ける

## 段階的リリース案

- Phase 1: 20 曲上限 + 1 曲 1 解説 + 手動停止
- Phase 2: 先読み生成（次曲分）とキュー可視化
- Phase 3: 高度制御（loop, 優先権, 途中再開, 管理UI）

## 判断メモ

- 機能としては実現可能。
- 成功の鍵は「同期競合のルール化」と「AI コスト制御」。
- まずは小さく始めて、運用ログを見ながら拡張するのが安全。

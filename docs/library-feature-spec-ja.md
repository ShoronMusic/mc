# 洋楽チャット「ライブラリ」機能 — 仕様確認（ドラフト）

本書は **実装前の合意形成用** のドラフトです。確定事項・未決事項を分け、既存の DB・API・管理画面案との整合を示します。

---

## 1. 背景と目的

### 1.1 現状

アプリの主導線は **YouTube URL を貼ることから開始** する（`AGENTS.md` 参照）。これに加え、次の利用者を想定する。

- URL の入手や YouTube 検索に不慣れな初めての利用者
- **特定アーティストの曲を連続で聴きたい**利用者

### 1.2 ライブラリで提供したい価値

- **DB に既に蓄積された曲**（主に `songs` / `song_videos` 経由で把握できるマスタ＋紐づく `video_id`）を、迷わず一覧・再生できる入口にする。
- 運営側では **管理画面から整備・追加入力** ができ、一般利用者向け UI は段階的に拡張できるようにする（**第1段は管理画面** とする）。

---

## 2. 情報設計（ナビゲーション）

### 2.1 第1軸: アーティスト別（既定）

- 分類キー: 既存の `songs.main_artist`（正規化ルールは既存の選曲・マスタ登録フローに合わせる）。
- **アルファベット INDEX** 一覧の例（表示イメージ）:
  - `(P) The Police (10)`  
    - 先頭の `(P)` は **頭文字インデックス**（The の扱いは別節）。
    - `(10)` は **当該アーティストに紐づく曲数**（`songs` 行数。同一 `display_title` の重複はマスタ設計に従う）。

### 2.2 第2軸: 固定タグ（年代 × スタイル等）

例: 「90s オルタナティブロック」のように **運営定義のタグ** で束ねる。

- **未決（要合意）**: タグのマスタをどこに持つか。
  - 案 A: `library_fixed_tags`（`slug`, `label_ja`, `description`）＋ `song_library_fixed_tag_assignments`（`song_id` または `video_id` FK）
  - 案 B: 既存の `songs.style` と年代列のみでフィルタ（タグ名の自由度は下がる）
- 年代の定義: **作品の原盤年** vs **YouTube クリップ公開年** で数値がずれるため、一覧の「公開年」表示と整合させる必要あり（**第4章**）。

### 2.3 第3軸: 自由タグ（任意グループ）

例: 「クリスマスソング」「ライブ定番」など、運営が任意に付与・編集するラベル。

- **未決**: 1曲に複数タグ可とするか、タグ階層（親子）を要するか。

### 2.4 検索窓

- 曲名・アーティスト名・（将来）タグ名の **部分一致検索**。
- 管理画面の既存 `/admin/songs` 検索（`/api/admin/songs-search`）を **拡張または再利用**する方針でよいか要確認。

## 3. 一覧表示（アーティスト配下の曲リスト）

### 3.1 表示カラム（要件より）

| 列 | データソース（案） | 備考 |
|----|-------------------|------|
| 公開年 | **要決定** | `songs.original_release_date` が無い場合は `song_videos.youtube_published_at` の年、または Music8 / 手入力。 |
| アーティスト | `songs.main_artist` | |
| タイトル | `songs.song_title` または `display_title` | UI は読みやすさ優先で `display_title` でも可。 |
| スタイル | `songs.style`（なければ `song_style` の代表 `video_id`） | 既存のスタイル解決と矛盾しないようにする。 |
| YT リンク | `song_videos` の **代表 `video_id`** | 複数動画があるときの「代表」のルール要決定（例: `variant='official'` 優先、無ければ最古 `created_at`）。 |
| 再生数（任意） | **下記 3.2** | 取れれば表示し、ソート可能にする。 |

### 3.2 ソート

- **既定**: 公開年月の **新しい順**（降順）。
- **任意**: 再生数の多い順。

**再生数の定義（要合意）**:

| 指標 | 意味 | 既存の近いもの |
|------|------|----------------|
| 曲単位（マスタ） | 「このチャットで同一作品としてカウントされた再生・選曲回数」 | `songs.play_count`（`docs/supabase-songs-and-performances-tables.md` で定義。`room_playback_history` 連動で増分される想定） |
| 動画単位 | 特定 `video_id` が流れた回数 | `room_playback_history` を `video_id` で `COUNT(*)` |

一覧は **曲（`songs`）1行** を主とするなら、ソートの主は `songs.play_count` が自然。列として **「代表動画の累計」** と **「曲マスタ累計」** のどちらを出すかは UI 文言で区別するか、片方に統一するか要決定。

## 4. 公開年・「公開」の語義

利用者向け文言が「公開年」の場合、次のどれを指すか明記する。

1. **音楽作品としてのリリース年**（シングル／アルバム原盤）
2. **YouTube 上でこのクリップが公開された年**
3. **マスタに手入力した表示用の年**

管理画面のライブラリでは、データが欠ける曲が多い想定のため、**欠損時のフォールバック順**（例: 1 → 2 → 「—」）を仕様で固定しておくことを推奨する。

## 5. 管理画面（第1実装スコープ）

### 5.1 配置と権限

- パス案: `/admin/library`（親）＋ `/admin/library/artists` または既存の `docs/admin-song-dashboard-spec.md` の **`/admin/artists` 案と統合`** するか要決定。
- API は `/api/admin/library/**` を新設する想定。
- 認証: 既存管理画面と同様 **ログイン必須**。`requireStyleAdminApi` 等、**誰が操作できるか**（STYLE_ADMIN のみか、広い admin か）を既存ポリシーに合わせて決める。

### 5.2 画面要素（要件より）

1. **アルファベット INDEX**（アーティスト名＋曲数）
2. **検索窓**
3. **URL 入力欄**（YouTube URL / `video_id` 相当）からの **曲追加**
4. アーティスト選択後の **曲一覧**（3.1 の列＋ソート）

### 5.3 URL から追加するフロー

1. URL から `video_id` を抽出（既存の YouTube パース処理を再利用）。
2. **メタデータ解決**: 既存の `resolveArtistSongForPackAsync` 系・oEmbed 等（`AGENTS.md` の `youtube-artist-song-for-pack`）と **同じ正規化** を通し、`songs` / `song_videos` を upsert。
3. **モーダル**を開く:
   - **埋め込みプレビュー再生**（管理画面内で問題ないか: 自動再生ポリシー・Cookie 等は既存プレイヤー実装に合わせる）。
   - **AI による解説（最大 5 本）**  
     - 既存ルーム選曲と同様、**comment-pack 相当**（基本 1 ＋ 自由最大 4 ＝計 5）を想定する。  
     - コード上の定数: `COMMENT_PACK_SOURCES`（`ai_commentary`, `ai_chat_1` … `ai_chat_4`）／`COMMENT_PACK_MAX_FREE_COMMENTS`（`src/lib/song-tidbits.ts`）。
4. **DB 保存**（フロントのルーム選曲と同様）:
   - `song_commentary`（`video_id` 単位の基本ライブラリ、`commentary-library.ts`）
   - `song_tidbits`（上記 5 スロット）
   - 必要に応じて `songs` の `style` 更新や `song_style` キャッシュ
   - Gemini 利用ログは既存どおり `gemini_usage_logs`（`context` は管理由来であることが分かる値を検討）

**未決**:

- 管理画面からの生成は **`POST /api/ai/comment-pack`** をそのまま呼ぶか、内部サービス関数を直叩きするか（レート制限・課金・ログの一貫性）。
- ルーム文脈（`recentMessages`）が無い場合のプロンプト差分（`COMMENT_PACK_SESSION_CONTEXT=0` に近い扱いでよいか）。

### 5.4 既存ドキュメントとの関係

- `docs/admin-song-dashboard-spec.md` … アーティスト INDEX・日別一覧・曲詳細の **全体像**。本ライブラリ機能はこれを **実装優先度の高いサブセット＋タグ＋URL 追加** として位置づけられる。
- `docs/supabase-songs-and-performances-tables.md` … `songs` / `song_videos` / `play_count` の意味。

# マイリスト（ユーザー機能）— 企画・設計メモ

**MusicAI チャットへの参加とは独立した**、ユーザー個人の「保存したい曲」の置き場。近い将来のマネタイズを見据え、**手放したくないほど使いやすい**ことを最優先する。

## 第1段階の必須スコープ（合意）

ライブラリを**アーティストを基軸に、曲がぶら下がる**考え方で育てる前提で、**第1段階では次を必須とする**。

| 必須 | 説明 |
|------|------|
| **項目・アーティスト** | 表示・検索・将来のアーティスト別一覧のたたき台。第1段階は DB の `artist` 文字列に永続化（§「アーティストの保存形態」）。**編集 UI で必ず確認・保存可能**。将来は **アーティストを DB 参照で読み込む**形へ移行。 |
| **項目・曲（曲名）** | 曲としての表示名。DB の `title`（表示用タイトル＝曲名）に永続化。**編集 UI で必ず確認・保存可能**にする。 |
| **動画の同一性** | `video_id` と `url` は必須（同一曲の識別・チャットへの選曲用）。 |

**第1段階で任意**: `note`、タグ、ジャンル、Music8 紐づけ、**アーティストの DB 参照（正規化）**（下記「目標」）。

## アーティストの保存形態（現状と目標）

**目標（ライブラリの考え方に合わせる形）**  
「保存のたびにアーティスト名をまた書き足すだけ」ではなく、**すでに DB にいるアーティスト（ユーザー単位または共有マスタ）を参照**し、マイリストの各曲は **そのアーティスト ID との紐づけ（多対多可）** でぶら下がる。**アーティスト別の件数・一覧**は、その参照（JOIN または集計）で読み出す。同一人物は **1 アーティスト 1 レコード**に寄せ、表示名の表記ゆれはマスタ側で解決する想定。

**現状（第1段階の実装）**  
`user_my_list_items.artist` に **表示用文字列**（複数は `, ` 区切り）を持つ**非正規化**のまま。これは **移行しやすいたたき台**であり、上記の参照モデルへ移すときは **マイリスト行＋中間テーブル**へデータを載せ替える。

**Supabase 追加テーブル（参照モデル）**  
`user_my_library_artists`（ユーザー別アーティスト行）と `user_my_list_item_artists`（曲とアーティストの多対多）。SQL は `docs/supabase-user-my-library-artists-tables.md`。**実装**: マイリスト `POST` / `PATCH` 成功後に `src/lib/my-list-sync-library-artists.ts` で `artist` 文字列と同期。

## 用語：マイリストとチャット内の「一時」との違い

| 概念 | 説明 |
|------|------|
| **マイリスト** | **自分のライブラリー**（永続）。DB に保存され、マイページで一覧・編集・削除する。 |
| **チャット参加中の一時** | **発言欄の下書き**、拡張で URL を**貼っただけ**の状態、**まだ送信していない**選曲 URL など。ブラウザ／セッション上の操作で、**ライブラリーには入らない**。 |
| **境界** | ライブラリーに載るのは、**「マイリストに保存」**や **マイページからの追加確定**など、ユーザーが**保存した**タイミングのみ。チャットで曲を**貼って再生しただけ**ではマイリストは増えない（従来どおり。貼った曲履歴・お気に入りは別テーブル）。 |

## 同一 `video_id`（確定方針）

- **1 ユーザーあたり、同一 `video_id` はマイリストに 1 行まで**（DB で `unique (user_id, video_id)` を推奨）。
- 別 MV（別 `video_id`）は別行。メモやタグで区別する。

## 自動入力の境界（詳細）

「何を**サーバーが勝手に DB に書いてよいか**」と「**提案まで**に留めるか」の切り分け。

### A. 確定保存まで自動でよいもの（低リスク・事実ベース）

| 処理 | 内容 | 注意 |
|------|------|------|
| URL 正規化 | `youtu.be` 等 → `watch?v=`、余計なパラメータ除去 | — |
| `video_id` 抽出 | 上記から確定 | — |
| oEmbed | YouTube が返す `title` / `author_name` を**初期値**として保存 API が書き込み可 | Topic チャンネル等で「アーティスト」がズレることがある → UI では編集可能に必ずする |

※ 初回 POST で `title`/`artist` を埋めるとしても、**ユーザーが後からいつでも上書き**できる前提。

### B. 自動で「提案」まで。DB の確定値にするのはユーザ操作があったとき

| 処理 | 内容 |
|------|------|
| 既存のタイトル分解 | `resolveArtistSongForPack` 系と同様に、**提案用**の `artist` / `song` を算出 | 保存 API が**無確認で上書き**しない。モーダルに「提案を反映」ボタン等。 |
| Music8（`videoId` 一致） | **参考パネル**に表示、またはフォームのプレースホルダ。**ワンクリックで反映**までに留めるのが無難 | 誤った JSON や古いデータのリスク。 |
| Music8（名前検索ヒット） | **参考のみ**。自動マージはしない |

### C. AI（Gemini 等）を使う場合

| 許容 | 禁止に近い |
|------|------------|
| **スタイル・年代・タグ案**を「AI 提案」ラベル付きで表示 | 根拠のないチャート順位・受賞歴などの**断定**（既存 `ai-output-policy` の精神に合わせる） |
| ユーザーが **「提案を採用」** を押した列だけ DB に書く | ユーザーが一度編集したフィールドを、**裏で次回保存時に AI が黙って上書き**しない |

### D. まとめ（実装ルールの一文）

- **機械取得（oEmbed）**は初期値として**保存してよい**が、**常にユーザー編集可能**。  
- **ヒューリスティクス・Music8・AI**は原則 **「提案」→ 明示的な採用**でライブラリーに混ぜる。  
- **チャットの一時状態**はマイリストの自動入力対象に**含めない**。

## 使用イメージ（目指す体験）

1. **YouTube の曲ページ**にアクセスする。
2. **起動**  
   - **PC**: Chrome 拡張（ツールバー等から）。  
   - **スマホ**: 専用ツールを起動する。**具体形は実装フェーズで決める**（例: OS の共有 → アプリ／PWA、ホーム画面ショートカット、将来のネイティブ等）。拡張と**同じく「確認してから確定」**の体験に揃える。
3. **モーダル**で保存内容（正規化 URL、`video_id`、自動取得したアーティスト・タイトル案など）を**確認し、保存を確定**する。
4. **MusicAI のマイページ**で **一覧の確認・修正・削除** を行う。
5. **別動線**: マイページから **URL を起点**に登録し、**AI やシステム**（oEmbed、既存の正規化、Music8 の `videoId` 一致時は**参考ヘルプ**のみ等）で基本情報を**提案**し、ユーザーが確定しながら項目を埋めていく。
6. **MusicAI 利用中に溜まったデータから**: マイページ **「貼った曲の履歴」** や **「お気に入りリスト」** の各行を起点に **マイリストへ追加**（内容確認のうえ保存）。チャットに**いま入っている必要はない**（履歴は過去の参加記録）。

チャット参加は**どちらの動線でも必須にしない**。

## 1. プロダクト上の位置づけ

| 観点 | 方針 |
|------|------|
| 主語 | **ユーザー**（各自のライブラリ。YouTube / Spotify でやっている「自分の棚」の近い体験） |
| チャット | **必須にしない**。部屋に入らなくてもマイリストを育てられる。 |
| Music8 | **必須にしない**。該当データがあるときだけ、項目やタグ付けの**参考**（提案・下書き）。 |
| 拡張 | **主要な入口の一つ**。YouTube 上で「正規化 URL をコピー → 発言欄」に加え、**マイリストへ直接保存**の動線を想定。 |
| チャット由来データ | **貼った曲**（`user_song_history`）・**お気に入り**（`user_favorites`）から **マイリストへ送る**動線を別途用意。役割が違うためテーブルは分けたまま、**UI で橋渡し**する。 |

既存のお気に入り（`user_favorites`）は **チャット視聴履歴起点**で設計済み。マイリストは **チャット外起点**を正式に扱うため、**別機能・別テーブル**で始めるのが安全（後から統合 UI は可能）。**貼った曲／お気に入りは「マイリストへの材料」**として再利用する。

## 2. ユーザーストーリー（例）

上記「使用イメージ」の具体例。

1. YouTube で曲を見つける → 拡張（または将来のスマホツール）のモーダルから **「マイリストに保存」**（ログイン必須）。
2. マイページで **一覧・検索・並べ替え**、または **URL 貼り付け → 提案の確認・確定** で追加。
2'. **貼った曲の履歴**／**お気に入り**の行から **「マイリストに追加」** → モーダルまたはインラインで内容確認 → 保存（既に同じ `video_id` がある場合はトーストやボタン無効で**二重登録を防ぐ**など）。
3. 各行は **アーティスト / タイトル** が一目で分かる（編集可）。
4. 余裕があれば **メモ・単一/複数タグ**、上級者向けに **ジャンル（プリセット or 自由）**。
5. 必要なときだけ、同じ URL を **チャット発言欄に送る**（現行拡張の「選択」動線と併存）。

## 3. Chrome 拡張との連携（設計）

現状（v0.1）：`MUSICAI_EXTENSION_SET_CHAT_TEXT_EVENT` で発言欄に URL を流し込む。

**マイリスト向け拡張案**（実装はフェーズ分割）：

| ステップ | 内容 |
|----------|------|
| B1 | モーダルに **「マイリストに保存」** を追加。対象は **洋楽チャット同一オリジン**のタブを開いたうえで、**API Route へ POST**（`credentials: 'include'`）。拡張から Supabase 直書きはしない。 |
| B2 | **チャットタブが無い**場合でも、拡張のオプションで許可した **マイページ／アプリのオリジン**にだけ POST できるようにする（`optional_host_permissions` 等。要プライバシー申告更新）。 |
| B3 | 保存直後に **toast / バッジ**で成功表示（任意）。 |

イベント名を増やす場合は `src/lib/musicai-extension-events.ts` と `service-worker.js` を**同一文字列**で同期すること（既存ルール）。

詳細の技術メモ: `docs/chrome-extension-musicaichat.md`（フェーズ B）。

## 3.1 マイページ：貼った曲・お気に入り起点（設計）

MusicAI チャットで過去に貯まった **自分のデータ**から、マイリストを育てる動線。

| 起点 | 主な取り込み元（既存） | マイリスト初期値の例 |
|------|------------------------|----------------------|
| 貼った曲 | `user_song_history`（`video_id`, `url`, `title`, `artist`, `posted_at`, `room_id`） | `url` / `video_id`、`title` / `artist` をコピー。任意でメモに「部屋 xx · 日時」 |
| お気に入り | `user_favorites`（`video_id`, `title`, `artist_name`, `display_name`, `played_at`） | `artist` ← `artist_name`、`title`、選曲者はメモ欄に `選曲者: display_name` 等（任意） |

**UI**: マイページ **曲管理**タブ内の「貼った曲」「お気に入り」リストで行ごとに **「マイリストに追加」**（ライブラリ本体は **トップレベルタブ「マイリスト」**）。押下後は **確認**（編集可）してから `POST /api/my-list`。

**`source` 列の拡張案**: `song_history` / `favorites` / `extension` / `manual_url` など。監査・将来の一括取り込み用。

**重複**: **ユニーク `(user_id, video_id)` 確定**（§「同一 video_id」）。API は二重 POST 時 **409** または **200 + 既存行そのまま（冪等）** のどちらかを選ぶ。

**実装メモ**: クライアントが `POST` に body で渡すだけでよい。サーバー側で `user_song_history` を再検証する必要は低いが、**改ざん防止で video_id が自分の履歴に存在するか**をオプションで検証してもよい。

## 4. データモデル（案）

**曲マスタ・アーティストマスタの項目（基本／拡張）**は `docs/song-artist-db-fields.md` を参照（マイリストはその薄いコピーとして設計）。

**新テーブル例: `user_my_list_items`**（名称は実装時に確定）

| 列 | 型 | 説明 |
|----|-----|------|
| `id` | uuid | PK |
| `user_id` | uuid | `auth.users` |
| `video_id` | text | YouTube video id（正規化） |
| `url` | text | 保存時の正規 URL（監査・共有用） |
| `title` | text | **曲名**（表示用。第1段階**必須**・ユーザー編集可） |
| `artist` | text | **アーティスト**（表示用。第1段階**必須**・ユーザー編集可。複数は `, ` 区切り等） |
| `note` | text | 任意メモ |
| `tags` | text[] または jsonb | 任意（段階導入） |
| `genre` | text | 任意（上級者・単一から開始可） |
| `source` | text | `extension` / `manual_url` / `song_history` / `favorites` / `import` 等 |
| `music8_song_id` | int null | 参考として紐づけた Music8 曲 ID（任意） |
| `created_at` / `updated_at` | timestamptz | 並べ替え・同期用 |

**制約**: **`unique (user_id, video_id)`**（§「同一 video_id」）。同一曲の**別 MV** は別 `video_id` で別行。

**保存時の自動入力**: §「自動入力の境界（詳細）」に従う。

## 5. API（案）

- `POST /api/my-list` … 追加（body: `videoId` または `url`、任意 `title`/`artist`/`note`/`tags`）
- `GET /api/my-list` … 自分の一覧（ページング・ソート）
- `PATCH /api/my-list?id=` … メタ編集
- `DELETE /api/my-list?id=` … 削除

認証は既存の Supabase セッションと揃える。

## 6. UI（案）

- **マイページ**の **トップレベルタブ「マイリスト」**（曲管理と同格）。
- **貼った曲の履歴**・**お気に入りリスト**（曲管理タブ内）の各行: **マイリストに追加**（§3.1）。
- 一覧: アーティスト・タイトル・追加日・（任意）タグ。フィルタは後から。
- 編集モーダル: 手入力しやすいフォーム。Music8 ヒット時は **「参考として反映」ボタン**程度。

## 7. マネタイズ（設計上のフックのみ）

仕様確定前の**方向性メモ**（実装コミットしない）：

- 無料枠: 件数上限・タグ数・エクスポート形式などを**後から**調整可能なように、API で `user_id` 単位の件数チェックを差し込める形にする。
- 価値の芯: **自分のデータが溜まる・直せる・持ち出せる**（CSV/テキスト等は信頼に直結）。

## 8. 実装フェーズ（推奨順）

1. **DB + CRUD API + マイページ最小一覧**（進行中）。§「第1段階の必須スコープ」どおり **`artist` / `title`（曲名）の管理・保存を必須**とする。テーブル SQL: `docs/supabase-user-my-list-table.md`。API: `src/app/api/my-list/route.ts`。マイページ **トップレベルタブ「マイリスト」**。
2. **貼った曲・お気に入りからの追加**（曲管理タブ内の行ボタン + `source` 区別）。
3. **拡張から POST**（B1: チャットタブ併用前提）。
4. **編集・タグ・ジャンル**の段階追加。
5. **Music8 参考 API**（保存時に任意でヒントを返す）。
6. B2（チャット無しでも保存）・ストア申告更新。

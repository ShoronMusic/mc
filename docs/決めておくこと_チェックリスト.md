# ここまでの段階で決めておくこと・チェックリスト

企画・仕様の議論を踏まえ、**すでに決まっていること** と **このあと決めておくとよいこと** を一覧にする。

---

## 1. すでに決まっていること（整理）

| 領域 | 内容 |
|------|------|
| **コンセプト** | AI と語る YouTube 同時視聴型・洋楽サロン。一人でも複数でも。同時利用（複数ルーム・金土夜のピーク）を想定。 |
| **差別化** | AI による登録ユーザー管理 ＋ 会参加時のホスピタリティ。一人利用では新曲開拓・発見とパーソナライズでリピート増。 |
| **認証** | 3種類：Google / 簡易会員登録 / ゲスト。登録でマイページ・振り返り・ユーザー館・個人化の対象に。 |
| **会話ログ** | 保存する。管理者が責任を持って管理。発言・参加・紹介曲・分析（user_ai_context）を user_id で紐づける。 |
| **DB** | Supabase（PostgreSQL）。テーブル設計は `DB設計_会話ログ保存版.md` の通り。 |
| **外部サービス** | Vercel, Supabase, Ably または Socket.io, Gemini API, YouTube IFrame API, oEmbed（曲メタ）。 |
| **AI の役割** | 曲解説（artist + title）、チャット参加（相槌・感想・盛り上げ）、入退室・曲切り替え時のホスピタリティ。個人化は user_ai_context をプロンプトに渡す。 |
| **フィードバック** | 会の満足度アンケート（満足度・AI 評価・自由記述）を保存し、サービス改善・AI 振る舞いの振り返りに使う。 |
| **MVP スコープ** | 1ルーム・1動画の同期再生 ＋ チャット ＋ AI 短い解説。設計は複数ルーム・同時利用を見据える。 |
| **サービス名** | **洋楽AIチャット**（仮）。正式決定までこの名称で進める。 |
| **ドメイン候補** | **musicaichat.com**。本番用として取得・使用を検討。 |
| **ルームの設置** | **登録ユーザーのみ**ルームを設置可能。ゲストは既存ルームへの参加のみ。 |

## 2. このあと決めておくとよいこと

### 2.1 開発着手前に決めるとよいもの

| # | 項目 | 内容・選択肢 | 備考 |
|---|------|----------------|------|
| 1 | **AI コンシェルジュの名前** | ユーザーに表示する名前・呼ばせ方（例: 「ラウンジの〇〇」）。 | プロンプト・UI の一貫性。 |
| 2 | **最初の画面のキャッチコピー** | トップや入室前の一言。 | 差別化の柱（覚えているAI・おかえり）を一言で伝える。 |
| 3 | **リアルタイムの選定** | Ably にするか、Socket.io（別サーバー）にするか。 | MVP では Ably の方が Vercel のみで完結しやすい。 |
| 4 | **MVP の「入れる機能」の線引き** | 認証は MVP でどこまでか（ゲスト＋Google のみか、簡易登録も含むか）。DB 保存は MVP からか。 | 構築手順とスケジュールに直結。 |

### 2.2 開発早期〜公開前に決めておくとよいもの

| # | 項目 | 内容・選択肢 | 備考 |
|---|------|----------------|------|
| 5 | **利用規約・プライバシーポリシー** | 会話ログ・発言分析・個人化の範囲・保存期間・オプトアウトを明文化。 | 登録ユーザー管理・個人化の前提。 |
| 6 | **モデレーション方針** | 荒らし・不適切発言への対応（削除・警告・BAN の基準、誰が対応するか）。 | チャットサービス詳細でレート制限・NGワードは触れ済み。運用ルールを固定。 |
| 7 | **会話ログの保存期間** | 本文を何年（または無期限）まで持つか。規約に書く数値。 | 運用・削除要請対応の基準。 |
| 8 | **ドメインの正式決定** | 候補 musicaichat.com を取得するか、別ドメインにするか。 | 公開時期と合わせてで可。 |

### 2.3 運用・成長に合わせて決めるもの

| # | 項目 | 内容・選択肢 | 備考 |
|---|------|----------------|------|
| 9 | **コスト・クォータの目安** | Gemini / Ably / Supabase の無料枠を超えそうになったときの目安やアラート。 | 金土夜のピークを見越した概算。 |
| 10 | **分析指標** | 何を測るか（DAU、セッション数、リピート率、アンケート平均、個人化オプトアウト率など）。 | 改善・差別化の効果測定用。 |
| 11 | **障害・フォールバック** | AI が落ちたとき・YouTube が埋め込み不可のときの表示・案内。 | ユーザー体験の下限を決める。 |
| 12 | **言語** | 日本語のみでいくか、多言語対応するか。 | UI・AI のプロンプト・規約に影響。 |
| 13 | **サービス名の正式決定** | 「洋楽AIチャット」のまま正式名とするか、別名に変更するか。 | ロゴ・商標・案内の確定時に検討。 |

## 4. まとめ

- **すでに決まっていること** は上記のとおり。仕様書群に落ちている。
- **このあと決めておくとよいこと** は、**開発着手前**（名前・コピー・技術選定・MVP 線引き）、**公開前**（規約・モデレーション・ログ期間・ルーム運用・ドメイン）、**運用後**（コスト・分析・障害時・言語）の3段階で整理すると進めやすい。
- サービス名は **洋楽AIチャット（仮）**、ドメイン候補は **musicaichat.com**、ルーム設置は **登録ユーザーのみ** で確定済み。特に **AI の名前・キャッチコピー** と **利用規約・プライバシー（個人化の説明とオプトアウト）** は、差別化と信頼に直結するので、早めに方針を固めておくとよい。

必要に応じてこのチェックリストを更新し、決まった項目は「すでに決まっていること」に移していく。

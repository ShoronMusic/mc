# 有料化・収益モデルの整理（案）

洋楽チャット（ルーム同期・YouTube・AI）向けの**代表的な課金パターン**と、**モバイル利用が増える前提**での決済の考え方。方針決定用メモ。

---

## 収益モデルの型

### 1. サブスク（月額／年額）

**例**: プレミアムルーム作成、AI 利用枠の拡大、広告なし、履歴・機能の解放。

| メリット | 収益が安定し、予測しやすい。 |
| デメリット | AI・外部 API・帯域の**原価がユーザーごとにバラつく**。**使い放題**にすると赤字リスク。 |

**運用のコツ**: 使い放題にするなら **「AI は月 ○ 回まで」**のように**上限付きサブスク**にしやすい。

---

### 2. チケット／クレジット制（回数券）

**例**: AI 曲解説パック N 回分、YouTube 検索ブースト N 回。

| メリット | **従量に近い原価**（Gemini・検索 API）と相性がよい。 |
| デメリット | 買い切り感が出るため、**リピート設計**（期限・お得パック）が要る。 |

モバイルでは小額の「ちょい買い」と相性がよい。

---

### 3. 従量課金（メーター課金）

**例**: 「AI ○ トークンあたり ○ 円」「検索 1 回 ○ 円」。

| メリット | 原価と収益の対応がきれい。 |
| デメリット | ユーザーが「いくらになるか分からない」と感じ**離脱**しやすい。請求・説明が難しい。 |

**現実的には**: 完全従量より **クレジットにパッケージ化**した方が運びやすいことが多い。

---

### 4. ハイブリッド（無料＋有料の組み合わせ）※現実的なことが多い

**例**:

- **無料**: 入室・再生・基本チャットは OK。
- **サブスク**: ルーム数・参加者上限・AI の「快適さ」・スタイルなど。
- **クレジット**: AI の追加・検索の追加・優先キュー。

**固定（サブスク）で LTV**、**変動（クレジット）で原価**を取りにいく形。

---

## モバイルが増えるときの決済

| 経路 | 内容 |
|------|------|
| **スマホのブラウザ（PWA / モバイル Web）** | **Stripe / PayPal 等の Web 決済**で済ませやすい。実装の自由度は高い。 |
| **ネイティブアプリ（App Store / Google Play）** | **アプリ内課金（IAP）**のルールが強い（デジタルコンテンツは特に）。手数料・審査あり。 |

よくある順序: **まずモバイル Web + Stripe** → 伸びたら **アプリ化 + IAP**。

---

## このプロダクト向けの相性（目安）

1. **無料 + サブスク（機能・上限付き）** … 収益の土台。
2. **+ クレジット（AI・検索の追加枠）** … 原価のブレを吸収。
3. **素直な従量の請求** … 上級者向け。一般向けは 2 に寄せた方が説明しやすい。

---

## 運用上の注意

- **AI・YouTube Data API・リアルタイム**は原価が動くため、「**無制限サブスク**」だけは慎重に。
- **利用規約**に「AI は誤る可能性」「外部 API 依存」などを明記しておくと安全。

---

## ゲスト向けの登録促進（嫌われにくい順）

ゲスト利用は一定数ある前提で、**体験を壊さずに本登録へ誘導**する案。  
基本方針は「即ブロック」より「便利さの差分」を見せる。

### 推奨（まず検討）

1. **ソフト導線（常時）**
   - 例: 「履歴保存・お気に入り・AI拡張は無料登録で解放」の固定導線をマイページ/投稿欄近くに表示。
   - 押し付け感が少なく、離脱を招きにくい。

2. **軽い回数上限（ゲストのみ）**
   - 例: 曲検索・AI質問・連続投稿の上限を会員より低くする（例: ゲストは 1 日 N 回）。
   - 使えるが、ヘビーユーザーだけ登録動機が生まれる。

3. **待ち時間差分（ゲストのみ）**
   - 例: ゲストは検索実行時に短いクールダウン、会員は即時。
   - 直接課金しなくても、体験差で自然に登録を促せる。

4. **機能差分**
   - 例: ゲストは閲覧・再生中心、会員はお気に入り、履歴の永続化、個人設定の保持を解放。
   - 「会員になる理由」が分かりやすい。

### 慎重（やりすぎ注意）

- **「CMを見ないと入室できない」強制視聴**
  - 初回離脱率が上がりやすく、コミュニティ型サービスでは嫌われやすい。
  - 使うなら「毎回」ではなく、頻度をかなり低くする。

- **画面占有型広告の多用**
  - モバイルで誤タップ・可読性低下が起きやすい。
  - 今の UI（チャット/プレイヤー中心）との相性は低い。

### 推奨しない組み合わせ

- 入室前の強制動画広告 + 高頻度ポップアップ + 機能制限の三重苦
  - 短期収益が出ても、継続率と口コミを損ねやすい。

---

## 実装するならの段階案（スモールスタート）

1. **Phase 1**: ソフト導線 + 会員メリットの明示（広告強化なし）
2. **Phase 2**: ゲストのみ軽い回数上限（検索/AI）
3. **Phase 3**: 反応を見て、必要なら低頻度の導線強化（強制視聴は最後の手段）

計測指標（最低限）:
- ゲスト→登録 CVR
- Day1/Day7 継続率
- 広告導入前後の離脱率（特にモバイル）

---

## 実際に採用する 3 施策（絞り込み版）

迷ったら、まずこの 3 つだけ実施する。

### 施策 1: 会員メリットの常時表示（ソフト導線）

- 内容: マイページ・投稿欄付近に「登録で解放される機能」を常時表示する。
- 訴求例: 履歴保存 / お気に入り / AI 拡張 / 設定保持。
- 目的: 押し付けずに「登録する理由」を明確化する。

### 施策 2: ゲストのみ軽い回数上限（検索・AI）

- 内容: ゲストだけ、検索・AI の利用回数を会員より低く設定する。
- 目安: まずは「不便すぎない」水準（例: 会員の 30〜50% 程度）から開始。
- 目的: ヘビーユース層に自然な登録動機を作る。

### 施策 3: 会員限定の保存系価値（永続化）

- 内容: お気に入り・履歴・個人設定の永続化を会員中心に提供する。
- 目的: 単発利用はゲストのままでも、継続利用者が会員化しやすくなる。

### この 3 施策で「やらない」こと

- 入室前の強制動画広告（CM 視聴しないと入れない）
- 画面占有型広告の多用

理由: モバイルで離脱・不満が出やすく、初期フェーズではリスクが高い。

## サーバー・外部サービス固定費（計算メモ）

### 定義

ここでの固定費は、**月額または年額を12で割り、毎月ほぼ一定に見える支出**とする。次は**従量**のため固定費に含めない。

- **含めない（従量）**: Google Gemini API、Vercel／Supabase／Ably の**超過従量**、Stripe の決済手数料、YouTube Data API のクォータ超過など。

### 為替・丸め

- 換算: **$1 = ¥160**（下の「2026-04-21 暫定収支メモ」の AI 単価換算と同じ前提）。
- ドル建ては **100円単位で丸め**（概算であることを明示）。

### プロダクトで想定する基盤

`docs/外部利用サービス一覧.md` に沿い、**Vercel**（ホスティング）・**Supabase**（DB／Auth）・**Ably**（リアルタイム）が中核。実装では **Resend**（トランザクションメール）も利用する。認証メール等で**送信頻度が高くない**うちは **Free のまま**で固定費に乗せないのが現実的（枠を超えるまで Pro は不要。少ない通数で月額だけ先に払うと**費用対効果が悪い**）。Stripe は課金導入時の決済基盤として、多くの場合 **月額なし（成功報酬型）**。

### 「固定費 45,000円/月」（暫定収支メモ）との関係

下記の暫定シートでは **固定費 `45,000円/月`** を按分として置いている。上表の **C（およそ ¥1.2万/月）** を軸に、必要なら **D**、さらに次のような要素を足し込んだ**運用上のバッファ**として解釈できる。

- 各サービスの**月次平均的な超過**（トラフィック・メッセージ増）
- **予備環境**や **Vercel の追加 seat** など
- **その他 SaaS**（監視・ログ等。未使用なら実額で下げる）
- インフラ以外の**固定に近い支出の按分**

運用では **カード明細・各ダッシュボードの前月請求（税込）**で突き合わせ、`45,000円` を**実測の合計**に差し替える。

### 月次の更新手順（チェックリスト）

1. Vercel・Supabase・Ably・Resend の **前月請求額**を転記する。
2. ドメインは **年額÷12** を加える。
3. その合計を「固定費（実測）」として暫定収支メモの前提に反映する。

## 半期シミュレーション（有料300人・2026年10月達成案）

当面の目標: **2026年10月末で有料ユーザー300人**。**2026年5月から**サブスクを **月1,000円・月間300曲上限** としたうえでの**月次収支の再シミュレーション**（暫定）。実績の人数・単価・`r` で差し替える。

### 共通前提

| 項目 | 値 |
|------|-----|
| プラン（300曲/月・1,000円）の想定開始 | **2026-05**。**シナリオA**は4月も有料 **0人**。**シナリオB**は先行契約で4月から **30人** が課金対象 |
| 1人あたり月額課金（総額） | **1,000円/月** |
| 1人あたり利用上限 | **300曲/月**（上限まで使い切る想定＝ストレス試算） |
| 決済手数料 | **3.6%**（手取り **×0.964**） |
| 月次固定費 | **45,000円** |

### 原価ブレンド（整数クレジット 1/2 に対応）

- **2クレジット級（NEW極）** 1選曲あたり変動: **1.60円**（暫定収支メモどおり: AI + YouTube 見込み + 変動インフラ）
- **1クレジット級（DB極）** 1選曲あたり変動: **0.40円**（解説パックの **Gemini 実費をゼロ** とみなし、YouTube **0.15** ＋ 変動インフラ **0.25** のみ。クイズ等が毎回飛ぶ場合は上乗せして `r` または単価を更新）

月間 **300曲** のうち、**2クレジット級**が占める割合を **`r`**（0〜1）とすると、**1人あたり月次変動費**は:

\[
300 \times \bigl(r \times 1.60 + (1 - r) \times 0.40\bigr)
\]

**1人あたり月次限界利益（固定の前）** = `964 −` 上式。

| ケース | `r`（NEW級の割合） | 1人・月 変動費 | 1人・月 限界利益 |
|--------|---------------------|----------------|-------------------|
| **基準** | **0.30** | **228円** | **736円** |
| **最重** | **1.00**（全曲NEW級） | **480円** | **484円** |

### 人数の立て方

**期末の有料UU**を **2026-04末 → 2026-10末** の7か月で直線的に増やす（月末スナップショット）。

- **シナリオ A**: 4月末 **0人** → 10月末 **300人**（毎月 **+50人**）
- **シナリオ B**: 4月末 **30人** → 10月末 **300人**（毎月 **+45人**。先行で有料契約が付いている想定）

**月次損益（簡式）**: `期末有料UU × (1人あたり限界利益) − 45,000`。

## Geminiオフ無料開放と混在ルーム運用（重要メモ）

### 1) Gemini APIを使わない場合の無料開放可否

- 結論: **無謀ではない。実施可能。**
- 前提:
  - Gemini実費が消えるため、原価の主軸は `Vercel` / `Supabase` / `Ably` / ドメイン / 運用工数になる
  - 収支より先に、無料運用特有の負荷・不正利用対策がボトルネックになりやすい
- 無料開放時の最低対策:
  - レート制限（投稿・検索・入室頻度）
  - ゲスト上限（1日利用回数、同時接続）
  - 部屋単位上限（参加人数、連投）
  - 異常アクセス監視と遮断運用
- 推奨段階:
  - **Phase 1**: Geminiなし無料で拡大
  - **Phase 2**: AI機能のみ課金（重い原価のみ有料化）

### 2) 無料ユーザーと有料ユーザーが同居する部屋の進行

- 結論: **実装可能。**
- 基本方針:
  - 「部屋単位」ではなく**選曲アクション単位**でAI稼働を判定する
  - その曲を入れたユーザーの権限で `aiMode` を確定する
- 進行ルール案:
  - 無料ユーザーの選曲 → システム進行のみ（AI呼び出しなし）
  - 有料ユーザーの選曲 → AI進行あり（comment-pack / quiz / 次曲提案）
  - 同一ルーム内で曲ごとに混在可
- 実装上の要点:
  - 曲投入時点で `aiMode` をイベントに保存（`none` / `full` など）
  - API側でも `aiMode` を再検証し、無料曲にAI呼び出しが漏れないよう二重ガード
  - UIで「通常モード」「AIモード」を明示して体験差を可視化

### 3) 課金設計上の注意

- 「部屋に有料ユーザーが1人いるから全員AI可」は避ける（ただ乗りが発生しやすい）
- 判定は「その選曲者が有料か」を原則にする
- 将来の例外（主催者が部屋全体をAI有効化する等）は、別商品として分離する

## 無料体験枠の消費ルール（DB解説とAI生成で変動）

### 方針

- 同じ「1曲」でも原価が異なるため、無料体験の消費量を固定せず**コスト連動**にする
- 体験ユーザーの無料枠は「5曲固定」より、**5クレジット制**で管理する

### 暫定ルール案

- DB由来の既存解説（AI新規生成なし）: **0.5クレジット消費**
- AI新規生成あり（comment-pack / quiz / next-song-recommend含む）: **1.0クレジット消費**

### 整数クレジット（解説パック5本・2パターン）

1選曲あたりの **comment-pack（基本1＋自由4）の5本**をまとめて見たときの**境界の2ケース**（設計・シミュレーション用）。

| パターン | 条件（5本すべて） | 消費クレジット（整数） |
|----------|-------------------|------------------------|
| **DB極** | ライブラリ再利用のみに相当（`[DB]` 主体・新規生成なし） | **1クレジット** |
| **NEW極** | 新規生成が主（`[NEW]` 主体・フル生成コスト） | **2クレジット** |

- **従来の小数案（0.5 / 1.0）との関係**: 比は **1 : 2** で同じ。**0.5＝表示1**、**1.0＝表示2** とみなせる。体験枠の総量は、内部を小数のまま **5** にするか、表示を整数に揃えて **10**（＝旧5と同じ許容量）にスケールするかを選べる。
- **混在**（5本のうち NEW と DB が混ざる）: 実装ではスロット加重・切り上げ・しきい値などを別途定義。収支シミュレーションでは **「2クレジット級」の選曲が月間に占める割合 `r`** で加重平均する（下の半期シミュレーション）。

### 期待効果

- 体験ユーザーは低コストな曲をより多く試せる
- 高コスト処理（AI新規生成）の無料消費を抑制できる
- 月次予算キャップとの整合が取りやすく、赤字制御しやすい

### 運用注意

- UIに必ず消費量を明示する（例: 「この曲はDB解説のため0.5消費」）
- バックエンドで消費計算を単一化し、フロント表示と不一致を起こさない
- まずは `0.5 / 1.0` の2段階で開始し、実績を見て調整する

### 検討課題: 先行者メリットの設計

- 背景:
  - DB解説が増えるほど後発ユーザーの体験効率が上がるため、先行ユーザーの不公平感が出る可能性がある
- 課題:
  - 早期参加者に対して、どの形でメリットを返すかを定義する必要がある
- 候補案:
  - 初期ユーザーへ追加クレジット付与
  - 新規解説作成に寄与したユーザーへのボーナス
  - 先行ユーザー限定の期間特典（価格/クレジット/機能）
- 判断基準:
  - 月次コストを圧迫しないこと
  - 新規ユーザーとの公平感を壊しすぎないこと
  - 説明が簡単で運用しやすいこと

# 同時利用と「一人でAIと楽しむ」設計

洋楽チャットサービスとして、**一人のユーザーがAI相手に楽しむ**ことを想定し、**同時に多数の利用ができること**を望む方針を整理する。

---

## 1. 方針

| 項目 | 内容 |
|------|------|
| **一人＋AI** | 他の参加者がいなくても、**一人でAIとチャット・曲解説・同時視聴**で楽しめる利用形態を想定する。 |
| **同時利用** | 複数のユーザーが**同じ時間帯に、それぞれのセッション（ルーム）で**利用できる。誰か一人が使っている間も、別の誰かが別の「部屋」で利用できる。 |

→ **洋楽チャットサービスとして「同時利用できる」ことを前提に設計する。**

### 1.1 金曜夜・土曜夜のピーク想定

**金曜の夜・土曜の夜には、複数人が参加するチャットルームが同時にいくつも始まることが十分想定される。**

- 例: 20:00 頃に「90s Rock」「ブリットポップ」「総合ラウンジ」など、**複数の公開ルーム**が同時に開き、それぞれに複数参加者が入室する。
- 設計は **「複数ルームが同時にアクティブである」** ことを前提とし、金土のピーク時でも安定して動作するようにする（接続数・AI 呼び出し・DB 負荷を考慮する）。

---

## 2. 利用形態のイメージ

| 形態 | 説明 |
|------|------|
| **一人＋AI** | 自分だけのルーム（またはパーソナルセッション）で、AI とだけチャット・曲を貼って解説してもらう。プランでいう「Studio」「The Vault」に近い。 |
| **少人数＋AI** | 友達数人と1ルームに入り、AI が相槌・解説で参加。 |
| **大人数＋AI** | 公開ルームに多数が集まり、AI が進行・盛り上げ。 |

いずれも **「そのルーム＝1セッション」** が同時に **複数** 存在し、それぞれが独立したチャット・再生・AI を持つ。

---

## 3. 設計上の前提（同時利用を満たすために）

| 項目 | 内容 |
|------|------|
| **ルーム／セッションの単位** | 1ルーム（または1セッション）＝ 1つのチャット・1本の同期再生・1体のAI。これが **複数並列** で存在する。 |
| **チャネル分離** | リアルタイムのチャネルは **ルーム（またはセッション）ごとに分ける**（例: `room:{roomId}` または `session:{sessionId}`）。他ルームの会話・再生は見えない。 |
| **一人＋AI** | 「パーソナルルーム」または「ソロ用セッション」を1ユーザーに1つ割り当てる（入室時に自動作成 or 選択）。そのルームにはそのユーザーとAIだけがいる。 |
| **同時利用数** | 同時に存在するルーム／セッション数に上限を設けない（または十分大きな上限）。**金曜夜・土曜夜は複数の大人数ルームが同時に立ち上がる**ことを想定し、リソース（Ably／Socket の接続数・Gemini の呼び出し・DB）をピーク時に見積もる。 |

→ **最初から「1グローバルルーム」ではなく、「複数ルーム／複数セッションが並列」** である形を目標にする。金土の夜の同時多ルームは **想定シナリオ** として設計に含める。

---

## 4. MVP との関係

| 段階 | 内容 |
|------|------|
| **MVP** | 実装コストを抑えるため、まず **1ルームのみ** で「全員が同じ部屋」にしてもよい。その場合でも、**データ構造とチャネル設計は「ルームID／セッションID で分離できる」形**にしておく。 |
| **公開後** | **複数ルーム** または **パーソナルルーム（一人＋AI）** を有効にし、**同時利用** を提供する。 |

→ **設計は同時利用を見据え、MVP ではルーム数を1にしても、拡張時には設定やルーム作成で増やせるようにする。**

---

## 5. パーソナルルーム（一人＋AI）のイメージ

- **入室時**: 「みんなのラウンジ」か「自分だけの部屋（パーソナル）」を選べる。
- **パーソナルを選んだ場合**: そのユーザー専用の `room_id`（例: `personal:{userId}`）に対応するセッションとチャネルが作成または再利用され、そこにはそのユーザーとAIだけが参加する。
- **同時に**: 別のユーザーが別のパーソナルルームや公開ルームに入っているため、**同時利用**が成立する。

---

## 6. まとめ

| 方針 | 内容 |
|------|------|
| **一人でAIと楽しむ** | 他の参加者がいなくても、一人でAIとチャット・曲解説・視聴で楽しめる利用形態をサポートする。 |
| **同時利用** | 洋楽チャットサービスとして、同じ時間帯に **複数のセッション（ルーム）が並列で利用できる** ようにする。一人＋AI はその一形態（パーソナルルーム等）で実現する。 |
| **金土夜の複数ルーム** | **金曜の夜・土曜の夜は、複数が参加するチャットルームが同時にいくつも始まる**ことを十分想定する。複数の大人数ルームが並列で動く前提で設計・リソース見積もりを行う。 |
| **設計** | ルーム／セッション単位でチャネル・チャット・再生・AI を分離し、複数が同時に動く形で設計する。MVP でルーム数を1にしても、拡張で複数ルーム・パーソナルルームを有効にできるようにする。 |

これにより、「一人のユーザーがAI相手に楽しむ」ことと、「金土夜に複数の大人数ルームが同時に立ち上がる」ことを両立できる。

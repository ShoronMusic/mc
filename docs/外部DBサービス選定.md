# 外部データベースサービスの選定

Music8 Lounge で利用する外部DBサービスとして望ましい候補を整理する。

---

## 前提（このプロジェクトの要件）

| 要件 | 内容 |
|------|------|
| データ構造 | リレーション型（users, sessions, session_songs, chat_messages, survey_responses 等） |
| 認証 | Google / 簡易会員 / ゲスト。別サービスでも可。 |
| リアルタイム | チャット・同期再生は Socket.io または Ably で実装予定。DBのリアルタイムは必須ではない。 |
| コスト | MVP は無料枠で始めたい。 |
| ホスティング | フロントは Vercel 想定。 |

---

## 推奨：Supabase（PostgreSQL）

**現状の企画・DB設計（PostgreSQL 想定）とそのまま合うため、最も望ましい。**

| 項目 | 内容 |
|------|------|
| **種類** | PostgreSQL（リレational）。既存の `DB設計_会話ログ保存版.md` のテーブル定義をそのまま使える。 |
| **認証** | Supabase Auth で Google ログイン・メール/パスワードが可能。ゲストはアプリ側で識別するだけなら Auth 不要。 |
| **リアルタイム** | Realtime 機能でテーブル変更を購読できる。チャットを「Socket ではなく DB の変更で配信」する構成も可能（任意）。 |
| **無料枠** | 500MB DB、50,000 月間アクティブユーザー等。個人・スモールスタートで十分。 |
| **連携** | Vercel と相性が良い。Next.js からは `@supabase/supabase-js` で接続。 |
| **管理** | Web ダッシュボードでテーブル・SQL・ログ確認がしやすい。 |

**デメリット**  
- 認証を M8 と「同じ Firebase」に揃えたい場合は、Supabase は DB 専用にして Auth は Firebase のまま、という構成も可能。

**結論**  
- 新規プロジェクトとして **DB も認証もまとめて用意する**なら **Supabase 一択**でよい。  
- 既存の DB 設計・SQL を活かしやすく、運用・拡張の見通しが良い。

---

## 代替案

### Neon（PostgreSQL） / Vercel Postgres

| 項目 | 内容 |
|------|------|
| **特徴** | サーバーレス PostgreSQL。Vercel が提供する「Vercel Postgres」は Neon ベース。 |
| **向いている場合** | 「DB だけ借りたい」「認証は別（Firebase 等）で統一したい」とき。Vercel と一体化した開発がしやすい。 |
| **無料枠** | あり。小規模なら十分。 |
| **注意** | 認証・ストレージ・Realtime はない。それらは別サービスで用意する必要がある。 |

→ **認証も含めて一つのサービスに寄せたい**なら Supabase。**Vercel に寄せて DB だけ軽く借りる**なら Neon / Vercel Postgres。

---

### Firebase / Firestore

| 項目 | 内容 |
|------|------|
| **特徴** | NoSQL（ドキュメント型）。リアルタイムリスナーが標準。Firebase Auth とセットで使いやすい。 |
| **向いている場合** | M8 ですでに Firebase を使っていて「認証・DB を同じ Google 基盤に揃えたい」場合。 |
| **注意** | 現在の設計は **リレーション型（users → sessions → messages）**。Firestore ではコレクション・サブコレクションでモデリングし直す必要がある。クエリの自由度は PostgreSQL より制約が多い。 |

→ スキーマを NoSQL 用に組み直す手間を許容するなら選択肢になる。**新規でリレーションをしっかり組みたい**なら PostgreSQL 系（Supabase）の方が設計と一致する。

---

### PlanetScale（MySQL）

| 項目 | 内容 |
|------|------|
| **特徴** | サーバーレス MySQL。ブランチングでスキーマ変更の検証がしやすい。 |
| **向いている場合** | MySQL に慣れている、または既存の MySQL 資産がある場合。 |
| **注意** | 外部キーが使えない等の制約あり。認証・Realtime は別サービス。 |

→ PostgreSQL で設計済みなら、わざわざ MySQL に合わせて設計を変えるメリットは小さい。

---

## まとめ：何を使うのが望ましいか

| 優先度 | サービス | 望ましい条件 |
|--------|----------|----------------|
| **第一推奨** | **Supabase** | リレーション型の設計をそのまま使いたい。認証（Google・メール）もまとめて用意したい。無料枠で MVP を始めたい。 |
| 代替 | Neon / Vercel Postgres | DB のみ外部化し、認証は Firebase 等で統一したい。Vercel 中心で開発したい。 |
| 条件付き | Firebase / Firestore | M8 と認証・基盤を Google に揃えたい。その代わり、スキーマを NoSQL 用に組み直すことを許容できる。 |

**結論**: 外部のデータベースサービスとしては **Supabase（PostgreSQL）** を使うのが望ましい。既存のテーブル設計・認証要件・コスト・運用のバランスが取りやすい。

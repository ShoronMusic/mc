# フィードバックとサービス改善

参加したユーザーに**会の満足度アンケート**に答えてもらい、その回答をフィードバックとしてサービスの改善と **AI の振る舞いの振り返り** に活用し、サービスの質の向上を目指す。

---

## 1. アンケートの目的

| 目的 | 内容 |
|------|------|
| **会の満足度の把握** | その回の体験がどうだったかを数値・自由記述で集める。 |
| **AI の振る舞いの評価** | AI の相槌・解説・盛り上げが適切だったか、多すぎ・少なすぎはなかったかなどを聞く。 |
| **改善の根拠** | 集まった回答を分析し、運営・AI プロンプト・機能の見直しに使う。 |

---

## 2. アンケートのタイミング

- **会終了後** … 退室時またはセッション終了後に「この会はいかがでしたか？」と1回だけ表示。
- **任意回答** … 強制にせず、答えたい人だけが回答。スキップ可能にする。

---

## 3. アンケート項目（案）

### 3.1 満足度（必須 or 任意）

| 項目 | 形式 | 例 |
|------|------|-----|
| 会全体の満足度 | 5段階（1: 不満 〜 5: とても満足） | ラジオボタン |
| また参加したいか | 5段階 または Yes/No | ラジオボタン |

### 3.2 AI について（サービス改善・AI 振る舞いの振り返り用）

| 項目 | 形式 | 用途 |
|------|------|------|
| AI の参加のちょうどよさ | 5段階（少なすぎ 〜 ちょうどいい 〜 多すぎ） | AI の発言頻度・存在感の調整 |
| AI の解説の役立ち | 5段階 または 役に立った / どちらでもない / 不要 | 曲解説の長さ・トーンの見直し |
| AI への自由記述 | テキスト（任意） | 「もっと〇〇してほしい」等の具体例収集 |

### 3.3 自由記述（任意）

| 項目 | 説明 |
|------|------|
| 良かった点・改善してほしい点 | 会全体・UI・曲の流れ・チャットの雰囲気など、自由に記入。 |

---

## 4. データベース（保存する情報）

アンケート回答を 1 件 1 行で保存するテーブルを用意する。

### survey_responses（アンケート回答）

| カラム名 | 型 | 必須 | 説明 |
|----------|-----|------|------|
| id | UUID | ○ | PK |
| session_id | UUID | ○ | どの会についての回答か。FK → sessions.id |
| user_id | UUID | △ | 回答者（正規ユーザー）。FK → users.id |
| guest_id | UUID | △ | 回答者（ゲスト）。FK → guests.id |
| satisfaction_rating | integer | - | 会全体の満足度（1〜5） |
| would_join_again | integer | - | また参加したいか（1〜5 または 1/0） |
| ai_fit_rating | integer | - | AI の参加のちょうどよさ（1: 少なすぎ 〜 3: ちょうどいい 〜 5: 多すぎ） |
| ai_commentary_rating | integer | - | AI の解説の役立ち（1〜5） |
| ai_feedback_text | text | - | AI への自由記述 |
| general_feedback_text | text | - | 良かった点・改善してほしい点（自由記述） |
| responded_at | timestamptz | ○ | 回答日時 |

- **制約**: user_id と guest_id のどちらか一方のみ非 NULL（ゲストでも回答可能にする場合）。
- **インデックス**: session_id, responded_at。集計時は session_id や日付範囲で絞る。

---

## 5. フィードバックの活用（改善サイクル）

### 5.1 管理者が行う振り返り

| 実施内容 | 説明 |
|----------|------|
| **回答の集計** | 会ごと・週ごとに満足度・AI 評価の平均・分布を確認。 |
| **自由記述の確認** | AI や会全体についての要望・不満を読み、傾向をメモ。 |
| **AI の振る舞いの振り返り** | 「多すぎ」「少なすぎ」の比率が偏っていないか、解説が「役に立った」と感じられているかを確認。 |

### 5.2 サービス・AI への反映

| フィードバックの種類 | 改善アクション例 |
|----------------------|-------------------|
| 満足度が低い会が続く | 時間帯・曲の流れ・ルール・モデレーションを見直す。 |
| AI が「多すぎ」に寄る | チャット参加のトリガーを緩める（間引きを強く、沈黙時のみなど）。発言文の長さを短くする。 |
| AI が「少なすぎ」に寄る | 相槌・感想の頻度を少し上げる。沈黙検知の閾値を下げる。 |
| 解説が「役に立たない」が多い | プロンプトを修正（長さ・トーン・専門用語の量）。140文字に収めるなど制約を強める。 |
| 自由記述の傾向 | 「盛り上げすぎ」「解説が堅い」などキーワードを拾い、AI のペルソナ・プロンプトを調整。 |

### 5.3 運用の流れ（例）

1. **定期集計** … 週1回 or 月1回、アンケート結果を集計し、満足度・AI 評価の推移を確認。
2. **振り返り** … 悪化している指標や自由記述の傾向を洗い出し、原因候補（AI・会の運営・UI）をリストアップ。
3. **改善** … AI のプロンプト・トリガー・発言頻度、または会のルール・機能を変更。
4. **再測定** … 変更後の会で再度アンケートを集め、改善したかどうかを確認。

---

## 6. 注意事項

- **匿名性** … 集計・振り返りでは「誰が」ではなく「どの会で・どの傾向か」を主に見る。個人特定が必要な対応は最小限に。
- **保存期間** … 利用規約・プライバシーポリシーでアンケート回答の保存期間と利用目的を明示する。
- **管理者のみアクセス** … 回答データは管理者が責任を持って管理。本番DBへのアクセスは管理者に限定する。

---

## 7. まとめ

- 参加ユーザーに**会の満足度アンケート**（満足度・また参加したいか・AI のちょうどよさ・解説の役立ち・自由記述）に答えてもらう。
- 回答は **survey_responses** に保存し、**管理者が集計・振り返り**する。
- 結果を**フィードバック**として、**サービスの改善**と**AI の振る舞いの見直し**（プロンプト・頻度・トーン）に活かし、**サービスの質の向上**を目指す。

# 同一時間帯に複数チャット（複数ルーム）を開く場合

サイト公開後、**同じ時間帯に複数のチャット（ルーム）を開けるか**について整理する。

---

## 1. 現状の設計（MVP）

| 項目 | 内容 |
|------|------|
| **ルーム数** | **1ルームのみ**。全参加者が同じ1つのチャット・同じ1本の動画を共有する。 |
| **同じ時間帯** | その時間帯に開いているチャットは **1つだけ**。複数はない。 |

→ **MVP では「同じ時間帯に複数チャットを開く」ことはできません。** 常に1ルーム・1チャットです。

---

## 2. 公開後に「複数チャットを同時に開く」とは

意味としては次のようなイメージです。

- 例: **金曜の夜・土曜の夜**など、20:00〜22:00 の時間帯に、
  - **ルームA**「90s Rock」
  - **ルームB**「ブリットポップ」
  - **ルームC**「総合ラウンジ」  
  などが**同時にいくつも**開き、それぞれに複数参加者が入室する。

→ **同じ時間帯に、複数ルーム（＝複数チャット）が並列で存在する**形。金土夜に複数の大人数ルームが同時に始まることは **十分想定される** ため、設計・リソースはこのピークを前提とする（`docs/同時利用と一人でAIと楽しむ設計.md` 参照）。

---

## 3. 技術的には可能か

**可能です。** すでに DB とチャネル設計が複数ルームを想定できる形になっている。

| 要素 | 現状 | 複数ルーム対応時 |
|------|------|-------------------|
| **DB** | `sessions.room_id` がある（MVP は 'main' 固定） | ルームごとに `room_id` を分ける（例: 'rock90s', 'britpop', 'main'）。 |
| **セッション** | 1ルーム＝1セッション or 1日1セッション | 「同じ時間帯に開いているルーム」＝ルームごとに1セッション（またはルームごとに常時1セッション）。 |
| **リアルタイム** | チャネル `room:main` のみ | チャネルを `room:{roomId}` で分ける。ルームごとにチャット・再生が独立。 |
| **チャット** | `chat_messages` は `session_id` で紐づく | ルーム（セッション）ごとにメッセージが分かれる。 |

→ ルームを増やし、**ルームごとにチャネル・セッションを分ければ、同じ時間帯に複数チャットを開けます。**

---

## 4. 複数ルーム対応にする場合のポイント

| 項目 | 内容 |
|------|------|
| **ルーム一覧** | トップやラウンジ入り口で「今開いているルーム」を一覧表示。ユーザーが1つ選んで入室。 |
| **入室** | 選んだルームの `room_id` に対応するチャネル（例: `room:rock90s`）にだけ join。他ルームのチャット・再生は見えない。 |
| **ルームの設置** | **登録ユーザーのみ**ルームを設置可能。ゲストは既存ルームへの参加のみ（`docs/サービス基本情報.md` 参照）。 |
| **AI** | ルームごとに1体の AI が付く想定。ルームごとに解説・相槌を生成（同じ Gemini API で、ルーム単位でコンテキストを分ける）。 |
| **セッション** | 「今このルームで行われている会」を `sessions` で管理。同じ時間帯にルームA用・ルームB用のセッションが複数存在する形。 |

---

## 5. まとめ

| 質問 | 答え |
|------|------|
| **MVP（現状設計）で、同じ時間帯に複数チャットを開けるか？** | **開けません。** 1ルーム・1チャットのみ。 |
| **サイト公開後に、同じ時間帯に複数チャットを開けるようにできるか？** | **できます。** ルームを複数用意し、チャネル・セッション・UI をルーム単位に分ける拡張で実現可能。 |

**一人のユーザーがAI相手に楽しむ**ことと、**同時に多くの人が利用できる**ことを両立するため、洋楽チャットサービスとしては **同時利用（複数ルーム／複数セッションの並列）を望ましい目標**とする。設計はその前提で行い、MVP でルーム数を1にしても、拡張で複数ルーム・パーソナルルーム（一人＋AI）を有効にできるようにする。詳細は `docs/同時利用と一人でAIと楽しむ設計.md` を参照。

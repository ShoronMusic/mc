# 部屋向け「洋楽でワイワイ」ミニゲーム／クイズ企画プラン

継続的にアイデア・優先度・仕様判断を追記するメモ。**実装タスクではなくプロダクト／体験設計の置き場**。

## 1. 背景と目的

- **洋楽AIチャットの差分**：一人視聴より、**みんなで同期視聴しながら会話・選曲する体験**が核。
- **狙い**：選曲・通常チャット以外に、**洋楽を題材にした短時間の参加型コンテンツ**で初対面でも会話が生まれ、距離が縮まる状態を作る。
- **非目標（このメモのスコープ外でもよい）**：ソロ向け本格ゲーム、長時間大会運営の完全自動化。

## 2. 設計原則（初対面・部屋のテンポ向け）

| 原則 | 内容 |
|------|------|
| 参加しやすさ | 正解より**参加・笑い・雑談**を主役にする。外しても傷つかない設問・ルール。 |
| 時間 | **1問 30〜90 秒**目安。視聴の流れを止めすぎない。 |
| 役割 | 司会固定にしない。**出題／投票／選曲が順番に回る**とハードルが下がる。 |
| AI の役割 | **素材（ヒント・選択肢案・解説）**に寄せ、判定は**多数決・拍手・任意の司会**で揉めにくくする。 |
| 信頼・安全 | 煽り・マウントが出やすいルールは避ける。荒れ時の**スキップ／オフ**を想定。 |

## 3. アイデア一覧（実装・運用の重さが軽い順の目安）

### 3.1 次の一曲当て（連想・ビンゴ風）

- 出題者がヒントのみ（年代・ムード・「こういう夜」等）→ 各自が YouTube で候補を投げる → **盛り上がり／拍手**で採用、など正解不要でも可。
- **既存との相性**：選曲フローと直結。

### 3.2 「偏愛 1 曲」自己開示ゲーム

- 各自 1 曲＋理由は一言 → 他者は**質問 1 つまで**など制限 → 拍手で締め（賞は任意）。
- **狙い**：クイズより**対人の距離**が縮まりやすい。

### 3.3 年代・スタイル当て（粗い分類の投票）

- 短く聴いて「70s / 80s / …」や「シンセ pop / グランジ」等を**投票**。多少ブレても議論がコンテンツになる。

### 3.4 2 択／4 択トリビア（洋楽豆知識）

- 曲解説・豆知識の延長として見せやすい。
- **注意**：誤答リスク。**断定しすぎない設問**、出典が曖昧な領域は避ける／「たぶん」表現寄せ。

### 3.5 歌詞 1 行／ニュアンス当て

- 短い引用や和訳ニュアンスから曲名・アーティスト当て等。
- **注意**：**著作権・引用の扱い**、運用ルールが必要。

### 3.6 協力型プレイリスト・ミッション

- テーマ（雨の日、帰り道等）に沿って**各自 1〜数曲ずつ**足して完成。ゲーム感は薄いが一体感は出やすい。

## 4. 優先度メモ（現時点のたたき）

| 優先 | 案 | 理由 |
|------|-----|------|
| 高 | 3.1 次の一曲当て | 正解不要・選曲 UI と親和・誤情報問題が小さい。 |
| 高 | 3.2 偏愛 1 曲 | 初対面向けの対人効果が大きい。 |
| 中 | 3.3 年代・スタイル投票 | 全員参加・判定が楽・短時間。 |
| 要設計 | 3.4〜3.5 クイズ／歌詞系 | 面白いが誤答・法務・運用コストが上がる。 |
| 具体案化 | **§8 曲解説後クイズ（3択）** | 体験は明確。AI 品質・同期・DB が設計の中心。 |

## 8. 曲解説後クイズ（3択）— 具体案・AI 可否・課題

### 8.1 体験の流れ（案）

1. **曲解説（既存）が出終わったあと**、§8.1b の**出題ゲート**を通過した再生に限り、同じ曲の文脈で **3 択 1 問**を表示する（**毎回ではない**）。
2. **問題文・選択肢・正解インデックス**は原則 AI が生成（初回）。以降は **DB にストック**し、同じ曲では**別バリエーションを複数保持**（繰り返し感を減らす）。全問を毎回生成する必要はない。
3. 参加者は再生中に回答。集計として **その回の正答率**（例：部屋内の集計）を表示する。
4. **ログインユーザー**は、回答・正誤・部屋／曲との対応を **参加履歴として保存**（マイページや統計の拡張を想定）。
5. **曲終了の数秒前**に **正解表示**と同時に **その回の正解者の表示名**を発表（拍手・演出は任意）。

### 8.1a 曲解説との役割分担（設問はサウンド・認知系を優先する案）

- **曲解説**で既に触れる概要・事実（アルバム名、年代の文脈など）と、**クイズの設問を被らせない**方が、1 曲あたりの情報が分散せず体験としてすっきりする。
- その代わり、クイズは **サウンド・認知系**（イントロの楽器や質感、テンポ感、ボーカルとバンドの関係、「この曲ではこう聴こえる」系）を**主軸**にするのが自然。
- **中リスクを許容する条件**：正解・誤答選択肢の根拠を **当該セッションの曲解説テキスト**（および曲解説生成時に渡した **同一の事実ブロック**／`facts_for_ai` 相当）**の範囲に閉じる**。つまり「曲解説に書いていないのにクイズだけで新事実を言う」を禁止し、幻覚余地を減らす。
- **表現**：必要なら「よく言及される」「この曲解説では」など**断定をやや弱める**語感で出題し、正解オープン時に一句だけ補足して締める、という逃げ道も検討可。

### 8.1b 出題の可否（毎回ではない・除外・オフィシャル限定の案）

- **毎回は出さない**：常時クイズだとリピート感とコストが増える。**確率抽選**／**直近 N 曲に 1 回まで**などのクールダウン／**部屋オプション**で「たまに出る」前提でもよい。
- **クイズを出さない／スキップしてよい例**：
  - **情報が少ない新曲**（曲解説・`facts_for_ai` 相当・マスタ等の**コンテキストが薄い**）
  - **マイナー曲**（同上、または外部参照が弱く信頼度が読めない）
  - **出どころが怪しい動画**（ファン再投稿・無関係チャンネル・権利・メタデータが信頼しづらい PV 等）
- **オフィシャル／信頼度の見分け（実務でよくやる方式）**：YouTube Data API や oEmbed だけでは **「公式 PV」と断定する万能フラグは期待しにくい**。その代わり、**取得済みの `channelId`・`channelTitle`・動画 `title`（必要なら `description` 冒頭）**から **ヒューリスティクスでスコアリング**するのが現実的でよい。
  - **プラス材料の例**：チャンネル名が **VEVO**、**`… - Topic`**（YouTube Music 系）、**レーベル／アーティスト名＋ Official**、既知の公式チャンネル **ID ホワイトリスト**に一致、など。
  - **マイナス材料の例**：チャンネル名が **Cover／Reaction／Fan／歌ってみた** 系、明らかに楽曲と無関係な投稿者、**公式っぽい語だけがタイトルに付いている**がチャンネルは素人、など。
  - **出力**：例として **`allow_quiz` / `uncertain` / `deny`** の三値。**`uncertain` はクイズ無し**（曲解説は別ゲートのまま）に寄せると安全。
- **厳しめ運用**：上記スコアが **`allow_quiz` に届いた再生だけ**クイズ対象、も選択可。
- **方針**：**誤出題より未出題**。ゲート落ちは黙ってスキップし、曲解説・再生体験はそのまま。
- **実装の寄せ道**：既存の **メタデータ品質・宣伝っぽさ**で AI 系をスキップしている経路（例：`shouldSkipAiCommentaryForPromotionalOrProseMetadata`）と **共通の「信頼度スコア」**を持てると、曲解説・クイズの挙動が揃いやすい。

### 8.1c 難易度（3段階）とオーナーによる出題レベル

- **難易度はおおよそ 3 段階**（例：**やさしい**／**ふつう**／**がっつり** ※表示ラベルはプロダクトで調整可）に分け、**生成プロンプト**と **DB ストック問のタグ**の両方に反映する。
- **チャットオーナー（部屋ホスト）が、会話の様子から判断してレベルを決める**のは面白く、初対面メインの部屋とマニア寄りの部屋の両立にも効く。**主観でよい**（自動推定は MVP では不要。将来、**直近の部屋内正答率**から「上げる／下げる」を**提案**する程度は可）。
- **段階のイメージ（案・§8.1a と両立）**：
  - **やさしい**：曲解説で触れた表現に寄せ、誤答肢が**聴感・語感ですぐ別物**に見える程度まで素直に。
  - **ふつう**：サウンド・認知の聴き分けを**一歩だけ**紛らわしく。
  - **がっつり**：**根拠は同一コンテキスト内のまま**、語彙の細かさや選択肢の紛らわしさだけ上げる（新事実を増やして難しくするのは避ける）。
- **DB**：ストック行に `difficulty`（1〜3 等）を持たせ、抽選時は **部屋に設定されたレベル**を優先し、足りなければ隣段階から補完する、などのルールが組める。
- **UI 案**：部屋設定の固定ドロップダウン、またはホストだけ「**次の曲からこのレベル**」の短いトグル（ライブ中に会話が盛り上がってきたら上げる、等）。

### 8.2 「AI が問題を考えられるか？」— 結論と条件

**結論：可能だが、「そのまま鵜呑み」は危険。設計で誤答リスクを下げる必要がある。**

| 観点 | メモ |
|------|------|
| 技術 | 3 択 JSON（問題文・3 選択肢・correctIndex・短い解説）は **モデル出力として十分現実的**。 |
| 品質 | 幻覚（存在しないメンバー名、年号、チャート順位等）が最大リスク。既存の **根拠の薄い断定を抑えるポリシー**（`ai-output-policy` 系）と同種の設計が必要。 |
| 推奨の土台 | 出題時に **既にアプリが持っている事実ブロック**だけをコンテキストに渡す（例：YouTube メタ、Music8 曲 JSON、`facts_for_ai` 相当、MusicBrainz 等で取れた確度の高い項目）。**コンテキスト外のタネネタは出さない**ようプロンプトで縛る。 |
| 設問タイプ | **曲解説直後のクイズ**は §8.1a のとおり **サウンド・認知系を優先**し、根拠は **曲解説と同一コンテキスト内**に閉じる。別用途のストック問として「年・アルバム」等の純事実 3 択を載せる場合は、**DB／マスタ照合可能**なものに限る。チャート 1 位・売上・噂話は避ける。 |
| フォールバック | 生成がポリシーで弾かれた・パース失敗した場合は **DB 既存問のみ**、なければ **その回はクイズなし**（視聴体験を止めない）。 |
| 人の目 | 初期は **STYLE_ADMIN や内部のみ「問題の品質報告」**で悪問を蓄積し、禁止パターンを増やす運用が現実的。 |

### 8.3 データ・同期まわりの検討メモ

- **ストック DB**：曲 ID（内部の song／video と紐づくキー）× **複数クイズ行**（`question_json`、**`difficulty`（§8.1c）**、`source`、`verified` フラグ、`usage_count`、`last_shown_at` 等）。抽選時に「最近出した問を避ける」「未使用を優先」、および **部屋の難易度設定**との整合。
- **その回のセッション**：`room_id` + `playback_event_id`（または `video_id` + 開始時刻）で回答を紐づけ、正答率・正解者リストを確定。
- **同期**：「曲終了の数秒前」は **クライアントの再生位置**と **サーバ時刻＋動画長**の両方でズレが出うる。MVP は **サーバが「正解オープン時刻」を配信**し、クライアントはそのイベントで UI を出す／Ably でブロードキャスト、などが無難。
- **プライバシー**：正解者名は **公開プロフィールの表示名規則**に合わせる。未ログインは匿名 or 不参加のみ、など方針を決める。

#### 8.3.1 現状実装と `room_chat_log`（チャット永続化）

同期部屋（`RoomWithSync`）では、画面上の `messages` と同じものが **`useRoomChatLogPersistence` → `POST /api/room-chat-log` → `room_chat_log`** に、**約 5 分間隔・タブ非表示時など**でバッチ保存される（`song_tidbits` 等の **別テーブルへの自動 INSERT ではない**）。

| 種類 | `room_chat_log` に載るか | 備考 |
|------|---------------------------|------|
| 曲解説・豆知識などの **AI 行**（`[NEW]` / `[DB]` 等） | **はい**（`message_type: ai`、本文） | 従来どおり。 |
| **【クイズ結果】…** の **AI 行**（正解・正解者の発表） | **はい**（`message_type: ai`、本文） | 他の AI 行と **同じ保存経路**。曲解説と **同一レコードに外部キーで束ねる**わけではなく、**会話の時系列**として並ぶ。 |
| **三択のシステム行**（見出し「三択クイズ（…）」＋ UI 用 JSON） | **一部** | 保存 API に送るのは主に **`body` 文字列**（短い見出し文）。**設問・選択肢・correctIndex の JSON は現状ログ行に含めない**。 |
| **各参加者の回答**（どの選択肢を押したか） | **いいえ** | Ably の `song-quiz:answer` のみ。§8.1 案の「参加履歴として回答を保存」は **未実装**。 |

まとめると、**結果発表の文面は曲解説と同様にチャットログ DB に残る**が、**クイズ本体の構造化データや回答ログを DB にまとめて保存する**ところまでは、現状の MVP では行っていない。

### 8.4 オープン論点（決めると実装が楽になる）

- [ ] 1 曲につき **同時に複数問**か、**常に 1 問**か。
- [ ] 曲解説とクイズの **トークン／API コスト**（曲ごとに毎回生成しない前提は §8.1 と整合）。
- [ ] 正解者「発表」が **全員表示**か **本人だけ通知**か（マウント感のトレードオフ）。
- [ ] **信頼度ゲートの定義**（§8.1b）：チャンネル名・タイトルパターンのルール一覧、**`channelId` ホワイト／ブラックリスト**、手動例外、スコア閾値（`allow` / `uncertain`）。
- [ ] **情報十分性の閾値**（`facts_for_ai` の有無・長さ、Music8 ヒット、MusicBrainz 等の最低条件）。
- [ ] **毎回ではない**の具体（出題確率、N 曲に 1 回、ユーザー／部屋のオプトアウト）。
- [ ] **難易度 3 段階のラベル**（ユーザー向け文言・英語 UI 時の表記）。
- [ ] **ホスト権限**：オーナー不在時、共同ホスト、ゲストにレベル変更を許すか。
